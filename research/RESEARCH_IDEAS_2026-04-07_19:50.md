## Idea 1: Ensemble Knowledge Distillation — Soft Targets from 16-Seed Ensemble

**What it is:** Use the 16-seed ensemble predictions (p_oodc=6.6, p_re=5.8 — both better than single-model) as soft training targets for a new single-model run, following offline distillation from ensemble.

**Why this is different from PR #2090 and #1982:**
- PR #2090 (knowledge distillation) closed: used the THEN-current ensemble as targets, but the ensemble was weaker and the single model had no TE coord frame, wake deficit, or PCGrad 3-way. The model being trained has changed dramatically since then.
- PR #1982 (offline distillation multi-seed): also pre-TE-frame, pre-wake-deficit. Results were mixed — "remarkable single model" but ultimately not merged.
- The KEY difference: the 16-seed ensemble currently beats the single model on p_oodc (6.6 vs 7.643) and p_re (5.8 vs 6.300) by substantial margins. These are now specifically the gaps worth closing.

**Mechanism:** Precompute ensemble predictions offline for all training samples. Use a blended loss: α * L1(pred, hard_gt) + (1-α) * L1(pred, ensemble_pred). The ensemble soft targets encode the average generalization across 16 random seeds — they are smoother, lower-variance versions of the ground truth that can regularize overfitting.

**Specific implementation:**
- Use `--ens_distill_weight 0.3` (30% soft, 70% hard): start with this ratio
- Load pre-cached ensemble predictions from `.npy` file for the training split only
- Blend is only on the surface loss (where ensemble advantage is largest)
- The soft-target file must be precomputed before training (run inference on the 16 models first)
- Blending ratio α=0.7 is based on the classic distillation literature (Hinton et al.) — temperature=1 (no logit softening needed for regression)

**Why it should work now:** The ensemble's advantage on p_oodc (+13.4% better) and p_re (+7.9% better) represents genuine information not in the single-model training signal. These improvements are specifically the OOD and Re-generalization gaps that remain after all our single-model improvements. Soft targets from the ensemble compress this information into a form the single model can learn.

**Expected impact:** -5 to -10% p_oodc, -3 to -7% p_re (the metrics where ensemble dominates). p_tan likely flat or slight improvement.

**Confidence:** MEDIUM. PR #1982 showed distillation can work but had instability. The stronger ensemble (16 seeds vs 8) and the better base model (TE frame + wake deficit) both increase the signal-to-noise ratio of the distillation target.

## Idea 2: Stochastic Depth with Late Warm Restart — Epoch 80+ Layer Drop

**What it is:** Apply layer dropping (stochastic depth / DropPath) selectively during the SECOND HALF of training (epochs 80-160), after the model has learned the base distribution.

**Why this is different from PR #2192 (stochastic depth, failed) and PR #2099 (DropPath, failed):**
- Both prior experiments applied stochastic depth uniformly from epoch 0, which disrupts early feature learning.
- The key insight from #2254 (backbone hidden noise) being in-flight: the CURRENT RESEARCH STATE document says "spectral norm SRF proved OOD failure is in the BACKBONE representation." Stochastic depth applied LATE acts as a regularizer on already-learned features — it should force the backbone to maintain redundant representations without disrupting initial convergence.
- This is the "late augmentation annealing" principle applied to architecture (analogous to #2255 aug annealing which turns augmentation OFF late — this turns dropout ON late).
- In curriculum learning literature (Bengio 2009), hard regularization early prevents learning. Hard regularization late (after convergence) forces the model to build redundant pathways — exactly what's needed for OOD.

**Specific implementation:**
- `--stochastic_depth_rate 0.1` — 10% drop probability for the first (earliest) TransolverBlock
- `--stochastic_depth_start_epoch 80` — only activate after epoch 80
- Each of the 3 blocks gets an independent drop rate: block 1 = 0.1, block 2 = 0.05, block 3 = 0.0 (output block never dropped)
- This is compatible with torch.compile (static shape — the block output is zeroed via masking, not removed)
- Memory overhead: zero (same graph, just zeroed outputs with a Bernoulli mask)

**Why late application:** The first 80 epochs are used to learn the basic physics. From epoch 80, stochastic depth forces the FIRST two blocks to learn representations that don't over-rely on each other — building implicit redundancy. This is exactly the mechanism that would help OOD: if block 1 "fails" (is dropped), block 2 must have already built enough representation to compensate.

**Expected impact:** -2 to -5% p_oodc, p_tan likely flat. The risk is p_in regression if the model loses convergence quality.

**Confidence:** MEDIUM-LOW. Prior attempts failed, but both applied stochastic depth from epoch 0. The epoch-offset curriculum is the distinguishing factor. The risk is that 10% drop rate is still destabilizing even at epoch 80.

## Idea 3: Target-Space Whitening — Predicting Standardized-Per-Foil Pressure

**What it is:** Instead of predicting the global-distribution-normalized pressure residual, predict a per-sample, per-foil standardized pressure. Each foil's pressure distribution is normalized to zero mean / unit variance before computing the loss, then de-normalized at inference.

**Why this hasn't been tried:**
- PR #597 (per-sample std normalization) was merged early but skipped tandem to preserve dual-foil learning.
- PR #2068 (asymmetric surface loss) and #2108 (asymmetric asinh scales) tried output-space modifications but not per-foil normalization.
- The current `--asinh_pressure --asinh_scale 0.75` compresses the GLOBAL distribution. But each foil has its own pressure regime — aft-foil in tandem operates in a lower-pressure recovery region with different dynamic range from fore-foil.
- PR #2136 (per-foil physics normalization, failed): tried to fix the Umag denominator, but this is different — it normalizes the TARGET distribution per foil, not the input normalization.

**Mechanism:** For each surface node, identify which foil it belongs to (using boundary IDs). Compute the per-foil mean and std of the pressure residual over each training sample. Normalize the prediction target by this per-foil distribution. At inference, de-normalize using the per-sample per-foil stats.

**Why this should help p_tan:** The tandem transfer failure mode is that the aft-foil pressure distribution is OOD (NACA6416 creates a different wake). Per-foil normalization removes the absolute pressure level and forces the model to predict the SHAPE of the pressure distribution relative to each foil's mean — a much more transferable representation. The model no longer needs to get the absolute pressure level right; it only needs to get the distribution shape right.

**Specific implementation:**
- `--per_foil_pressure_norm` flag
- Per-sample normalization: for each sample, compute `p_foil_mean` and `p_foil_std` per boundary-ID group (IDs 5,6 = fore-foil surface; ID 7 = aft-foil surface)
- Clamp std to minimum 0.05 to avoid division by zero
- Apply before asinh: normalize → asinh → predict
- De-normalize at inference: un-asinh → rescale by std → add mean
- The normalization stats are computed per-sample during the forward pass, not precomputed

**Key risk:** If the per-foil std is too small (smooth pressure distributions), the normalization amplifies noise. The 0.05 clamp mitigates this.

**Expected impact:** -3 to -8% p_tan (the OOD foil shape transfer gap). Likely modest p_oodc improvement too.

**Confidence:** MEDIUM. The physical motivation is strong (foil-specific pressure normalization is aerodynamically natural), and the mechanism is distinct from all prior target-space changes.

## Idea 4: Backward-Compatible Curriculum via Sample Difficulty Replay

**What it is:** During training, maintain a "hard sample buffer" — samples where the EMA model's prediction error exceeds a threshold. Every K epochs, replay hard samples at 2x frequency by injecting them as duplicates in the dataloader without modifying the sampler structure.

**Why this is different from existing hard-node/sample ideas:**
- PR #2101 (OHEM) and #2257 (focal sample reweighting, in-flight): these reweight the LOSS, not the sampling distribution. The model still sees each sample once per epoch.
- Hard sample REPLAY changes the gradient budget — the model makes 2x gradient steps on hard samples per epoch.
- The key insight: `--focal_gamma=0.5` (in-flight) modulates gradient magnitude. Sample replay modulates gradient FREQUENCY. These are orthogonal mechanisms.
- PR #1981 (error-based curriculum, failed): tried curriculum ORDERING (easy-to-hard over epochs), not replay. The failure was that easy-to-hard ordering hurts: it effectively reduces the training distribution early.

**Specific implementation:**
- After epoch 30 (after initial convergence), at the START of each epoch:
  1. Run a mini-evaluation pass on the training set with NO_GRAD to get current EMA model errors
  2. Select samples with surface pressure MAE > 1.5x the median as "hard"
  3. Add these samples as duplicates to the current epoch's dataloader (they appear twice)
- `--hard_replay_after_epoch 30` — enable after epoch 30
- `--hard_replay_threshold 1.5` — 1.5x median threshold
- `--hard_replay_max_frac 0.3` — at most 30% of the epoch budget is replay (prevents collapse to tiny hard set)
- Only applies to the surface loss computation for replayed samples (volume loss unchanged)

**Why the no-grad evaluation pass is feasible:** The training loop already does val every 2 epochs. A fast train-set pass using EMA model adds ~15 seconds per epoch — acceptable given the 180-min budget.

**Critical detail:** The hard sample buffer must be re-evaluated periodically (every 5 epochs) to track which samples are still hard. Stale buffers reinforce the same patterns without adaptation.

**Expected impact:** -3 to -6% p_tan (tandem hard cases are consistently hard across epochs), -2 to -4% p_oodc.

**Confidence:** MEDIUM. Replay buffers are standard in reinforcement learning (Experience Replay, Prioritized Experience Replay — Schaul et al. 2016) and curriculum learning. The direct analogy is PER (Prioritized Experience Replay) applied to supervised learning.

## Idea 5: Wake-Relative Attention Bias — Physics-Prior Slice Routing for Tandem

**What it is:** Add a physics-informed attention bias to the slice routing logits in Transolver: nodes that are geometrically "in the wake" of the fore-foil get a learnable bias toward specific slices. This is a targeted inductive bias that encodes aerodynamic prior knowledge directly into the attention mechanism.

**Why this is distinct from all prior spatial bias work:**
- PR #2130 (gap/stagger spatial bias): adds gap/stagger SCALARS to the existing spatial_bias MLP. The MLP learns global conditioning.
- This adds a STRUCTURED bias based on the wake-deficit feature (already computed as `--wake_deficit_feature`): nodes with large wake_deficit magnitude (i.e., deep in the fore-foil wake) get routed toward different slices than nodes outside the wake.
- The wake deficit feature (gap-normalized fore-TE offset: dx/gap, dy/gap) is ALREADY IN THE FEATURE VECTOR — this idea uses it to construct an attention prior, not just a feature.

**Mechanism:**
- Compute `wake_score = torch.norm(wake_deficit_features, dim=-1, keepdim=True)` — scalar per node encoding "how deep in the wake is this node?"
- For tandem samples: add `wake_score * learnable_wake_bias_vector` to the slice logits before softmax, where `learnable_wake_bias_vector` is a learned R^slice_num vector initialized to zero
- For single-foil samples: wake features are already zeroed (by the existing `--wake_deficit_feature` implementation), so this bias is exactly zero — no effect on single-foil samples
- Initialization: zero → identical to baseline at epoch 0. The model learns to USE the wake prior only if it helps.
- Parameter count: 96 additional scalars (one per slice). Negligible.

**Physical motivation:** In tandem configurations, the inter-foil channel nodes and the aft-foil leading edge nodes are the hardest to predict. They are also the nodes with the highest wake_score. By allowing the model to route wake-affected nodes to dedicated slices, we give it the opportunity to specialize a subset of slices for "wake physics" — without forcing it to do so.

**Expected impact:** -2 to -5% p_tan. The wake deficit feature (PR #2213) already helped p_tan -0.6% as a plain feature. Turning it into a routing prior exploits the same signal at a different level of the architecture.

**Confidence:** MEDIUM. The zero-init safety is high. The mechanism is orthogonal to all current features and loss functions. The main risk is that the slice routing is already well-optimized and the additional bias adds noise.

**Specific implementation flag:** `--wake_routing_bias` with no hyperparameters needed beyond the flag (all initialized to zero, learned from data).

## Idea 6: Tandem-Specific EMA Decay — Slower EMA for OOD Tracks

**What it is:** Maintain TWO EMA moving averages: one with the standard decay=0.999 (for p_in, p_re checkpointing) and one with a slower decay=0.9995 or 0.998 that has more weight on earlier training epochs. The slower-decay EMA is used for p_tan and p_oodc checkpoint selection.

**Why this hasn't been tried:**
- PR #2244 (higher EMA decay 0.9995, FAILED): applied uniformly to ALL metrics. The failure was that 0.9995 helped OOD but hurt p_in because it averaged over earlier, worse p_in predictions.
- The KEY insight: different validation tracks benefit from different EMA timescales. p_in benefits from the most recent weights (high decay = fast EMA). p_oodc and p_tan benefit from MORE averaging (lower decay = slower EMA that includes older, more-generalized weights).
- This is related to Stochastic Weight Averaging Dense (SWAD, PR #2094, failed) which densely averaged checkpoints but across the wrong time horizon.

**Mechanism:**
- Maintain two EMA copies: `ema_fast` (decay=0.999, current) and `ema_slow` (decay=0.997)
- At validation time, evaluate BOTH EMA models on each val track
- For checkpointing: use `ema_fast` for p_in and p_re; use `ema_slow` for p_tan and p_oodc
- During training, use `ema_fast` for gradient updates (the model the student trains on). The `ema_slow` is purely a checkpoint candidate.

**Why `ema_slow` should help OOD:** The OOD validation tracks (p_tan, p_oodc) are testing generalization. The slow EMA has higher weight on weights from 20-40 epochs earlier in training, when the model may have broader representations before overfitting to in-distribution patterns. This is the "model soup at a discount" idea — instead of averaging discrete checkpoints, we continuously average with a longer time horizon.

**Memory overhead:** One additional copy of model weights (~75 MB at n_hidden=192). Negligible on 96GB GPUs.

**Specific implementation:**
- `--dual_ema --ema_slow_decay 0.997`
- Both EMAs start at the same start_epoch (current default ~epoch 140)
- Final checkpoint selection: min across both EMA models per metric

**Expected impact:** -2 to -4% p_tan, -1 to -3% p_oodc. p_in likely unchanged (uses fast EMA).

**Confidence:** MEDIUM. The key insight that different validation tracks may benefit from different EMA timescales has not been tested before. The prior failure of uniform 0.9995 EMA is specifically explained by the "one EMA for all tracks" limitation.

## Idea 7: Chord-Relative Pressure Gradient Prediction — Change the Target

**What it is:** Instead of predicting absolute pressure residuals, predict the GRADIENT of pressure along the chord direction (dp/ds) as an auxiliary target, and reconstruct the pressure via cumulative integration. The model predicts both the mean pressure and the gradient field simultaneously.

**Why this is distinct from prior target-space changes:**
- PR #1867 (pressure_first): separates pressure and velocity prediction pathways — still predicts absolute p.
- PR #2186 (panel Cp residual, failed): tried predicting deviation from panel method inviscid Cp, but the panel method had poor tandem coverage.
- PR #2183 (vorticity auxiliary, failed): used vorticity as an AUXILIARY target, not the primary pressure target.
- The insight from aerodynamics: pressure GRADIENTS along the surface are the physically meaningful quantities (they determine lift via integration). The leading edge suction peak is a sharp feature in p but a structured peak in dp/ds. The neural network's spectral bias makes it harder to learn sharp absolute values than smooth gradient fields.

**Mechanism:**
- For each surface foil (sorted by arc-length s), compute `dp_ds[i] = (p[i+1] - p[i]) / (s[i+1] - s[i])` at node level
- Add auxiliary loss on `dp_ds` prediction: `L_grad = 0.05 * L1(pred_grad, true_grad)` on surface nodes only
- The model predicts BOTH `p` (absolute, as now) and `dp_ds` (gradient, new head)
- The gradient head is a small 2-layer MLP of hidden=64 attached to the SRF output
- The gradient target normalizes the learning signal around sharp LE/TE transitions

**Key difference from DCT loss (PR #2184, merged):** DCT loss operates in frequency domain globally. Gradient loss is in spatial domain locally, specifically at adjacent node pairs on the surface arc. DCT loss is complementary (frequency domain), while this provides spatial-domain gradient supervision.

**Expected impact:** -2 to -5% p_tan (LE suction peak is the hardest feature on NACA6416, and it's a local gradient feature). The gradient auxiliary loss explicitly teaches the model where pressure changes sharply.

**Confidence:** MEDIUM-LOW. The fundamental target-space insight is sound (pressure gradient is more physically structured than absolute pressure), but the auxiliary loss weight needs careful tuning. The DCT loss already provides some frequency-domain supervision — overlap risk with `dct_freq_loss`.

**Specific implementation:**
- `--pressure_grad_loss --pressure_grad_weight 0.05`
- Arc-length computation: use cumulative Euclidean distance between consecutive surface nodes after sorting by arc-length position
- Initialize gradient head weights near zero to avoid regression at start of training

## Idea 8: Condition-Aware Batch Normalization Stats — Separate BN for Each Flow Regime

**What it is:** Replace `DomainLayerNorm` (which already uses domain-conditional normalization based on tandem vs. single-foil status) with a stronger version that uses FOUR separate normalization stat sets: (1) single-foil standard, (2) tandem normal Re, (3) tandem extreme AoA, (4) extreme Re. Each condition computes its own running mean/variance statistics and its own affine parameters.

**Why this is distinct from PR #1800 (DomainLayerNorm, merged):**
- DomainLayerNorm uses 2 sets of gamma/beta: single-foil vs tandem. The same 4-way split that PCGrad uses (#2119, merged: tandem normal, tandem extreme, single) is NOT reflected in the normalization.
- The insight: PCGrad already knows which samples belong to which regime (via `--pcgrad_extreme_pct 0.15`). The normalization should also respect this.
- 4-way condition-aware normalization has been shown effective in domain adaptation literature (Conditionally Adaptive Multi-Task Learning, ICLR 2021) — using per-condition BN stats allows each domain to have its own activation scale without gradient interference.

**Specific implementation:**
- `--quad_domain_norm` — 4-condition LayerNorm replacing the current 2-condition DomainLayerNorm
- Conditions: determined by (is_tandem, is_extreme) where is_extreme uses the same `pcgrad_extreme_pct=0.15` threshold
- Each condition has its own gamma/beta vectors (4x the parameter count of DomainLayerNorm, still tiny)
- For the val splits: val_in_dist → condition 0, val_oodc → condition 2 (extreme), val_tandem → condition 1 (tandem normal), val_oodc_re → condition 3 (extreme Re)
- This is compatible with torch.compile: the condition index is a scalar, not a dynamic shape

**Why it should help p_oodc and p_tan:** The current DomainLayerNorm separates single vs tandem, but the 4 validation tracks test 4 DISTINCT flow regimes. The model uses the same normalization stats for both tandem-normal and tandem-extreme-AoA — these are physically quite different (extreme AoA cases have separated flow, very different activation magnitude). Separate normalization gives each regime its own internal representation scale.

**Memory overhead:** 4x DomainLayerNorm parameters (still ~16KB total). Zero VRAM impact.

**Expected impact:** -2 to -4% p_oodc (primary beneficiary: condition OOD split), -1 to -3% p_tan.

**Confidence:** MEDIUM. DomainLayerNorm was one of the impactful Phase 3 merges. Extending it from 2 to 4 conditions follows the same logic with finer granularity. The risk is that 4 conditions are too many for the training data — each condition sees fewer samples per batch, noisier normalization stats.
