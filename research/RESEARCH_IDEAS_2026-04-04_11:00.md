## Idea 1: Separate SRF Heads for Fore-Foil (ID=6) vs Single-Foil (ID=5)

**Priority: TOP. From the confirmed priority queue. ~30 LoC. Expected -3 to -6% p_tan. LOW RISK.**

**Hypothesis:**
The current surface refinement head (`surface_refine=True`) uses a single MLP applied to all surface nodes regardless of boundary identity (ID=5 single-foil, ID=6 fore-foil tandem, ID=7 aft-foil tandem). The dedicated aft-foil SRF branch (#2104, fern, in-flight) is being tested for ID=7. But the fore-foil (ID=6) is also aerodynamically distinct from the single-foil (ID=5): it experiences accelerated flow in the channel gap and modified lift characteristics due to the presence of the aft foil. A single shared SRF head is forced to serve all three surface domains with the same learned correction function, which is a representational bottleneck. Separate SRF heads per boundary allow each MLP to specialize to the correction patterns of its domain without interference.

**Implementation instructions:**
1. In `SurfaceRefinementHead.__init__`, replace the single `self.mlp` with three separate MLPs: `self.mlp_single`, `self.mlp_fore`, `self.mlp_aft`.
2. In `SurfaceRefinementHead.forward`, add a `boundary_id` parameter (integer tensor `[M]` of IDs per surface node: 5=single, 6=fore, 7=aft). Route each node through the appropriate MLP:
   ```python
   def forward(self, hidden, base_pred, boundary_id):
       inp = torch.cat([hidden, base_pred], dim=-1)  # [M, n_hidden+3]
       correction = torch.zeros_like(base_pred)
       for mlp, bid in [(self.mlp_single, 5), (self.mlp_fore, 6), (self.mlp_aft, 7)]:
           mask = (boundary_id == bid)
           if mask.any():
               correction[mask] = mlp(inp[mask])
       return correction
   ```
3. In the training/val loop, where surface nodes are extracted via `is_surface.nonzero()`, also extract the boundary ID. In `prepare_multi.py` (read-only), boundary ID is stored in `x[:, :, some_index]` — check the feature schema. From the code: `x[:, 0, 21]` is the gap feature (tandem discriminator). The boundary ID is likely in `x[:, :, some_feature]` before standardization — check `raw_dsdf` or the raw `x` for an integer boundary-ID feature. **Alternatively**: derive it from the gap feature for tandem discrimination: if `is_tandem` and `x_coord < midpoint`, it's fore-foil; if `is_tandem` and `x_coord >= midpoint`, it's aft-foil; else single. This avoids needing to read the raw boundary ID.
4. Zero-initialize all three MLPs' last layers (preserves the identity correction property).
5. Total parameter addition: 2x the current SRF head size (two extra heads; one replaces the original).

**Specific flags:**
```
--surface_refine True
--surface_refine_hidden 128
--surface_refine_layers 2
--surface_refine_separate_ids True   # new flag
```
Or simply hard-code the split without a new flag for simplicity.

**Expected magnitude:** -3% to -6% p_tan (fore-foil correction specialization mirrors what aft-foil SRF #2104 is expected to produce for ID=7). Low risk — zero-init preserves baseline behavior if the split is wrong.

**Risk:** LOW. Backward-compatible with zero-init. The only failure mode is if the fore-foil correction patterns are too similar to single-foil to benefit from separation.

## Idea 2: Charbonnier Loss — Ultra-Smooth L1 Variant

**Priority: HIGH. ~5 LoC. Natural follow-up after Huber (#2113 in-flight). Expected -1% to -3% across all metrics. LOW RISK.**

**Hypothesis:**
The Charbonnier loss `L(x) = sqrt(x^2 + eps^2) - eps` (Charbonnier et al., 1994) is differentiable everywhere unlike L1, and unlike Huber it has a smoothly varying gradient across all magnitudes with no threshold discontinuity. For mesh-based CFD surrogates, the gradient signal matters most near surface nodes where errors are small but the MAE metric is tight. The Charbonnier loss provides a continuously varying gradient (not a hard switch like Huber at `|x| < beta`) which may produce smoother convergence. With `eps=0.1`, it approximates L1 for large errors and L2 near zero — a different interpolation profile than Huber. Note: this is NOT the same as Huber (smooth L1). They have different gradient schedules.

**Implementation instructions:**
In the training loop, replace the abs_err-based loss computation with Charbonnier:
```python
# Current:
abs_err = (pred - y_norm).abs()

# Charbonnier replacement:
charbonnier_eps = cfg.charbonnier_eps  # e.g. 0.1
abs_err = (((pred - y_norm) ** 2 + charbonnier_eps ** 2).sqrt() - charbonnier_eps)
```
This is a drop-in replacement for `abs_err` — all downstream loss computation (vol_loss, surf_loss, hard-node mining, coarse pooling) automatically uses Charbonnier.

**New Config flag:**
```python
charbonnier_eps: float = 0.0   # 0.0 = standard L1 (no Charbonnier), >0 enables it
```

**Sweep values:** `eps = {0.05, 0.1, 0.2}` — `eps=0.1` is the standard starting point from the image restoration literature (Dosovitskiy, Brox ECCV 2016; Sun et al. CVPR 2010). Run 3 seeds per value.

**Expected magnitude:** -1% to -3% on surface MAE (based on image restoration improvement profile). Modest but additive if it stacks with current loss.

**Risk:** LOW. Reverts to L1 exactly when `eps=0`. Only downside is slightly more compute for the sqrt.

## Idea 3: Tandem-Specific PCGrad Weighting — Bias Group B Toward p_tan Splits

**Priority: HIGH. ~15 LoC modification. Targets p_tan directly. Medium confidence.**

**Hypothesis:**
The current PCGrad implementation splits batches into Group A (in-dist: single-foil, normal Re/AoA) and Group B (OOD: tandem OR extreme Re OR extreme AoA). When Group B contains a mixture of tandem, extreme-Re, and extreme-AoA samples, the gradient from tandem-foil errors is diluted by the gradient from extreme-Re and extreme-AoA errors within the same Group B backward pass. This means the PCGrad projection protects in-dist gradients from OOD *in aggregate*, not specifically from the tandem *sub-distribution* that dominates p_tan. The fix: split Group B further into Group B1 (tandem) and Group B2 (extreme-Re/AoA without tandem), compute three separate backward passes, and project all three against each other. This is a multi-task PCGrad extension. The key hypothesis: tandem-foil gradient interference with extreme-condition gradients is a meaningful noise source for p_tan improvement.

**Implementation instructions:**
```python
# Current: is_ood_pcgrad = is_tandem_batch | (re_ood) | (aoa_ood)
# New: three-way split
is_tandem_pcgrad = is_tandem_batch
is_re_aoa_ood = (x[:, 0, 13] > 1.0) | (x[:, 0, 14].abs() > 1.0) & ~is_tandem_batch
is_indist_pcgrad = ~is_tandem_batch & ~is_re_aoa_ood

# Only run if all three groups are non-empty
if is_tandem_pcgrad.any() and is_re_aoa_ood.any() and is_indist_pcgrad.any():
    # Three backward passes, project all three pairwise (PCGrad-3way)
    ...
# else: fall back to current 2-way PCGrad or no PCGrad
```

**Config flag:** `--pcgrad_tandem_split True` (new flag, default False).

**Expected magnitude:** -2% to -5% p_tan. The mechanism is theoretically sound but depends on tandem vs. extreme-Re/AoA samples actually generating conflicting gradients. If the groups don't actually conflict (their gradients are aligned), PCGrad reverts to averaging and there's no change.

**Risk:** MEDIUM. Adds a third backward pass when all three groups present; may increase per-epoch time by ~30% in the worst case. Add an early-out if any group is empty.

## Idea 4: Boundary-ID One-Hot as Sideband Feature — Explicit Surface-Type Conditioning

**Priority: HIGH. ~10 LoC. Novel surface-type conditioning. Expected -3% to -8% p_tan. Medium confidence.**

**Hypothesis:**
The model currently infers tandem vs single-foil through the gap feature (`x[:, 0, 21]` post-standardization), which is a scalar per-sample feature. For individual surface nodes, there is no explicit feature telling the model "this node is on the fore-foil of a tandem configuration" (boundary ID=6) vs "this is the inner surface of a single foil" (boundary ID=5). This per-node boundary disambiguation could give the model direct routing signal for the SRF head and attention slices, without relying on the model to infer node identity from positional/geometric cues alone. Adding a one-hot boundary-ID feature as a new input channel (3-dim: [is_single, is_fore, is_aft]) provides this signal explicitly at the node level.

From the code, `raw_dsdf = x[:, :, 2:10]` contains signed-distance field gradients. The boundary ID is implicitly encoded in the SDF channel selection (dsdf[:, :, 4:8] is the secondary foil SDF for tandem — already used by `foil2_dist`). However, a one-hot encoding is a cleaner signal that doesn't require the model to learn to distinguish this.

**Implementation instructions:**
In the training loop (before standardization), compute boundary one-hot:
```python
# Before: x = (x - stats["x_mean"]) / stats["x_std"]
# The boundary ID is derivable from existing features:
# - is_tandem: x[:, 0, 21].abs() > threshold (gap feature, same for all nodes in a sample)
# - is_surface: already available
# - For tandem samples, fore vs aft: use x-coordinate split (fore-foil is upstream)
#   Approximation: fore-foil nodes have x < 0.5 * (x_min + x_max) for tandem

# After standardization, append one-hot:
gap_raw = x[:, 0, 21]  # gap, pre-standardization (use before stats normalization)
is_tandem_node = (gap_raw.abs() > threshold)[:, None].expand(-1, N)  # [B, N]
# Crude fore/aft split by x-coordinate median per tandem sample
x_coord = x[:, :, 0]  # raw x before standardization
x_mid = x_coord.median(dim=1, keepdim=True).values
is_fore = is_tandem_node & (x_coord < x_mid)
is_aft = is_tandem_node & (x_coord >= x_mid)
is_single = ~is_tandem_node & is_surface  # only valid on surface; volume nodes get [0,0,0]

one_hot = torch.stack([is_single, is_fore, is_aft], dim=-1).float()  # [B, N, 3]
x = torch.cat([x_standardized, one_hot], dim=-1)
```

Update `fun_dim` in model config: `fun_dim += 3`.

**Note:** The x-coordinate split is an approximation. A cleaner version uses `is_surface` and the secondary dsdf sign to distinguish fore vs aft. Check if the raw dsdf gradient direction (indices 4-7, secondary foil) flips sign between fore and aft nodes — it should, since the aft foil's SDF is positive for fore-foil nodes (they're outside the aft foil's boundary).

**Expected magnitude:** -3% to -8% p_tan. The model can now explicitly condition surface-type decisions on a discrete per-node signal, which is far more informative than deriving it from geometric cues alone.

**Risk:** MEDIUM. The one-hot relies on the x-coordinate approximation being a reasonable fore/aft splitter. May not work if fore and aft foils overlap in x-coordinate space (tandem stagger configurations). Test with the simple version first; refine if needed.

## Idea 5: Langevin Gradient Noise — SGLD-Style Exploration for Flatter Minima

**Priority: MEDIUM-HIGH. From the confirmed priority queue. ~10 LoC. Expected -1% to -3% p_in. Medium confidence.**

**Hypothesis:**
Stochastic Gradient Langevin Dynamics (SGLD, Welling & Teh, 2011) injects Gaussian noise into gradient updates proportional to the learning rate, turning SGD into a Markov Chain Monte Carlo sampler that explores the posterior distribution of model parameters. For ML optimization (without the full Bayesian interpretation), the key effect is that noise injection at small scales during late training prevents the optimizer from converging to sharp narrow minima — it naturally finds flatter, better-generalizing basins. This is distinct from SAM (which destabilizes Lion, confirmed dead end): SGLD adds noise *after* the gradient update rather than perturbing parameters before the gradient step. It is also distinct from target noise (already tuned). With Lion (sign-based updates), gradient noise before the sign operation creates a stochastic perturbation to the update direction — a form of exploration that SAM cannot provide without perturbing the weight space.

**Implementation instructions:**
After `loss.backward()` and `clip_grad_norm_`, before `optimizer.step()`, add Gaussian noise to each parameter's gradient:
```python
if epoch >= cfg.langevin_start_epoch and cfg.langevin_noise_std > 0:
    noise_t = cfg.langevin_noise_std * (1 - (epoch - cfg.langevin_start_epoch) / max(MAX_EPOCHS - cfg.langevin_start_epoch, 1))
    for p in model.parameters():
        if p.grad is not None:
            p.grad.data.add_(noise_t * torch.randn_like(p.grad.data))
```

**New Config flags:**
```python
langevin_noise_std: float = 0.0    # 0.0 = disabled; try 1e-4 to 1e-3
langevin_start_epoch: int = 50     # start late to avoid disrupting warmup
```

**Suggested values:** `langevin_noise_std = {1e-4, 5e-4, 1e-3}`, `langevin_start_epoch = 50`.

**Key interaction with Lion:** Lion uses `sign(exp_avg * b1 + grad * (1-b1))` as the update direction. Gradient noise before this step perturbs the effective gradient direction, not just its magnitude — the sign operation makes this a discrete flip probability, which is a bounded and stable form of exploration.

**Expected magnitude:** -1% to -3% on p_in and p_oodc (flat-basin finding). p_tan benefit is more speculative.

**Risk:** LOW-MEDIUM. Noise std is a continuous dial — `std=0` reverts to baseline. Main risk is over-injection destabilizing late-stage convergence, but `langevin_start_epoch` parameter allows safe onset control.
