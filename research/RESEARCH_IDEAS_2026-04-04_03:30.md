## Ranked Hypotheses

### 1. Iterative / Recurrent Transolver (Fixed-Point Iteration)

**Priority: HIGH. From RADICAL_IDEAS Honorable Mention #9. Never assigned. Directly targets p_tan.**

The current Transolver runs the attention computation through 3 blocks sequentially with different weights. Instead, use K=3-4 weight-tied iterations over a single block — equivalent to unrolling a fixed-point iteration until convergence. This is the Implicit Layer / Deep Equilibrium Network (DEQ) approach (Bai et al., NeurIPS 2019), but applied without the expensive Jacobian-free fixed-point solver: just K forward passes through the same parameter-set block, treating the hidden state as the iterate.

**Physical analogy:** RANS pressure-velocity coupling is itself an iterative solver (SIMPLE/PISO). The model is learning to approximate the outcome of a convergent iterative process — giving it explicit iterative structure aligns the computational graph with the physics. The pressure field on tandem configurations requires long-range coupling between the fore-foil's wake and the aft-foil's suction peak: iterative refinement can propagate this signal more efficiently than a feedforward stack of distinct blocks.

**Why it might help:** p_tan = 29.1 is 2.5x worse than p_in = 12.2. The hypothesis is that the tandem OOD failure is a receptive-field problem: the 3-block architecture has limited range for fore-to-aft signal propagation. Fixed-point iteration with the same block run K=4 times effectively gives 4x depth without increasing parameter count. VRAM budget is unchanged or reduced (fewer unique block parameters = fewer optimizer states).

**Complexity estimate:** ~30-40 lines. Wrap the existing `TransolverBlock` call in a loop over K iterations, passing the hidden state as input to the next iteration. The crucial implementation details:
- Weight tying: pass the SAME block object K times (not K separate block instances)
- Hidden state initialization: zero or output of a lightweight learned linear projection
- Gradient through all K iterations (standard backprop, no fixed-point solver needed for K=4)
- Optionally apply LayerNorm to the hidden state between iterations

**Implementation sketch:**
```python
# In Transolver.__init__:
self.blocks = nn.ModuleList([TransolverBlock(...)])  # single shared block
self.n_iters = 4  # K

# In Transolver.forward:
fx = self.embedding(x)
for _ in range(self.n_iters):
    fx = self.blocks[0](fx, fx_q=fx)  # same block, updated hidden state
fx_deep = fx  # for deep supervision if also enabled
```

**Key hyperparameters:** K=3 or K=4 (start with K=3 since current 3-block cost ~ K=3 shared-block cost); learning rate may need slight reduction due to deeper effective gradient path.

**Expected impact:** p_tan primarily, potentially p_oodc. No change expected for p_in (single-foil already well solved).

**Key paper:** Bai et al., "Deep Equilibrium Models", NeurIPS 2019. For the lightweight unrolled version (no Jacobian-free solver): Fung et al., "Fixed Point Networks", 2021.

**Confidence:** Moderate-strong theoretical motivation. No CFD-surrogate validation, but the physical analogy is tight. The risk is gradient instability through K=4 unrolled iterations — use gradient clipping (already in train.py).

### 2. Precomputed Pressure-Poisson Soft Constraint

**Priority: HIGH. From RADICAL_IDEAS Honorable Mention #10. Never assigned. Distinct from failed WLS physics losses.**

The Poisson equation for pressure in incompressible flow is: Lap(p) = -rho * div(u x u) approximately -rho * div(u) * ||u|| (at mean-field level). The key insight that makes this distinct from the FAILED physics losses (#2016, #2023, which crashed due to WLS instability): compute the Laplacian stencil OFFLINE from the k-NN mesh graph and store as a sparse matrix. At training time, compute `Lap(p_pred)` using the precomputed stencil and penalize its deviation from the precomputed RHS from the training data.

**Why this is NOT the same as dead-end #2016/#2023:** The WLS-based losses (#2016, #2023) tried to compute differential operators via Weighted Least Squares at training time from the raw mesh coordinates — this was numerically unstable due to conditioning issues. The precomputed approach bakes the finite-difference stencil into a fixed sparse weight matrix saved to disk (one matrix per mesh topology). At training time it's just a sparse matmul — no numerical instability.

**Why it might help:** p_tan = 29.1 suggests the model is not enforcing pressure-velocity consistency on tandem configurations. A soft constraint that penalizes Poisson residuals provides a physics-derived gradient signal that the L1 surface loss cannot provide — the L1 loss only penalizes node-level errors, not divergence from physical consistency.

**Complexity estimate:** ~50 lines for offline stencil computation + ~15 lines for training loss addition.

**Implementation sketch:**
```python
# OFFLINE (precomputation, run once):
# For each mesh in dataset: compute k-NN graph, fit FD Laplacian stencil via LS on neighborhoods
# Save as torch sparse tensor: laplacian_matrix[mesh_id] shape [N, N]

# TRAINING (in loss computation):
L = precomputed_laplacian[batch_mesh_id]  # [N, N] sparse
lap_p_pred = torch.sparse.mm(L, pred[:, :, 2])  # Laplacian of predicted pressure
lap_p_true = torch.sparse.mm(L, target[:, :, 2])
poisson_loss = F.l1_loss(lap_p_pred, lap_p_true)  # soft: match true Laplacian pattern
loss = main_loss + 0.05 * poisson_loss
```

**Key hyperparameters:** Stencil neighborhood k=8-12 neighbors; loss weight 0.01-0.1 (start 0.05); apply only to volume nodes (not surface — surface values are what we care about, volume is auxiliary).

**Expected impact:** p_tan and p_oodc (OOD configurations where pressure-velocity coupling is most different from training distribution).

**Key paper:** Lino et al., "Towards Fast Simulation of Environmental Fluid Mechanics with Multi-Scale Graph Neural Networks", ICLR 2022. Mesh-based GNN with physics constraints via precomputed Laplacian.

**Confidence:** Moderate. The mechanism is sound and distinct from the failed WLS approach. The precomputation requires careful implementation of the FD stencil fitting. Start with the soft constraint only (no hard enforcement).

### 3. Tandem Fore/Aft Decoupled Loss Weighting

**Priority: HIGH. ~8 lines. Directly targets p_tan. Not the same as asymmetric pressure loss.**

The asymmetric pressure loss (previously suggested, not yet assigned to a student) penalizes under-prediction of suction peaks. This idea is orthogonal: give the AFT foil (boundary ID 7) a higher loss weight than the FORE foil (boundary ID 6) during training.

**Physical motivation:** The aft foil is aerodynamically harder because it operates in the wake of the fore foil. The model likely learns well for the fore foil (which is close to single-foil behavior) but struggles with the aft foil's modified pressure distribution. Upweighting the aft foil loss forces the model to specifically improve aft-foil prediction accuracy, which is the primary driver of p_tan regression.

**Note on distinction from dead-end Magnitude-Weighted Loss (#2068):** The magnitude-weighted loss (#2068) upweighted loss by the absolute magnitude of the predicted/true value (higher pressure = higher weight). The fore/aft decoupled weighting here is based on which physical component the node belongs to — a structural/topological criterion, not a magnitude criterion.

**Complexity estimate:** ~8 lines. Read the boundary ID from the batch, build a per-node weight tensor, apply to the L1 loss.

**Implementation sketch:**
```python
# In training loop (after target loading):
# boundary_id shape: [B, N], values include 6 (fore), 7 (aft), others (volume)
aft_mask = (boundary_id == 7)  # aft foil nodes
loss_weights = torch.ones_like(target[..., 0])
loss_weights[aft_mask] = cfg.aft_foil_weight  # new flag, default 2.0-3.0

per_node_loss = F.l1_loss(pred[surface_mask], target[surface_mask], reduction='none')
weighted_loss = (per_node_loss * loss_weights[surface_mask].unsqueeze(-1)).mean()
```

**Key hyperparameters:** `aft_foil_weight` in {1.5, 2.0, 3.0} — start with 2.0. Apply only during tandem training samples (when `tandem_ramp` logic is active or condition indicates tandem).

**Expected impact:** p_tan primarily. Might slightly worsen p_in if tandem weight is too aggressive (single-foil has no aft foil, so the mask is empty — no effect on single-foil samples).

**Confidence:** Strong conceptual motivation. The specific weight requires tuning. The implementation is trivial and the risk of regression on p_in is low (the mask is empty for single-foil samples).

### 4. Mesh-Density Weighted L1 Loss

**Priority: MEDIUM-HIGH. ~15 lines. Targets leading-edge and wake high-gradient nodes.**

In unstructured CFD meshes, the mesh spacing is finer near the airfoil surface (especially at the leading edge and trailing edge) and coarser in the far field. The current L1 loss treats each node equally. Weighting by `1 / local_mesh_spacing` (upweighting fine-mesh regions) makes the loss proportional to physical domain area rather than node count.

**Why it might help:** The model may be "smoothing over" sharp pressure peaks at the leading edge because the peak region has very few nodes (each node is precious there). By upweighting these nodes, we explicitly force the model to fit high-curvature regions. This directly targets the source of p_tan error: suction-peak underprediction.

**Complexity estimate:** ~15 lines. Compute local spacing as mean distance to k-nearest neighbors per node; normalize; use as loss weight.

**Implementation sketch:**
```python
# During data loading (precomputed or on-the-fly):
local_spacing = knn_distances.mean(dim=-1)  # [N] -- mean distance to k=4 nearest neighbors
density_weight = 1.0 / (local_spacing + 1e-6)  # [N]
density_weight = density_weight / density_weight.mean()  # normalize to mean=1.0

# In training:
per_node_loss = F.l1_loss(pred[surface_mask], target[surface_mask], reduction='none')
weighted_loss = (per_node_loss * density_weight[surface_mask].unsqueeze(-1)).mean()
```

**Key hyperparameters:** k=4 for local spacing estimate; weight normalization (mean=1 recommended to avoid changing effective LR); apply only to surface nodes. Optionally clip weights at 99th percentile to avoid extreme upweighting at singularities.

**Expected impact:** p_in and p_tan (both have sharp leading-edge gradients). Lower expected impact on p_oodc and p_re which are smoother.

**Key paper:** Alet et al., "Mesh-based simulation with deep learning", NeurIPS 2019. Uses per-node area/volume weighting in GNN physics simulations.

**Confidence:** Moderate. The mechanism is sound for unstructured mesh settings. The tradeoff between fine-mesh (high gradient) and coarse-mesh (far field) weighting needs careful calibration.

### 5. Periodic (Sin) Activation in Surface Refinement MLP Only

**Priority: MEDIUM-HIGH. ~8 lines. Not full SIREN -- only in the srf head.**

Full SIREN (Sinusoidal Representation Networks, Sitzmann et al. 2020) was tried as a full architecture replacement and was catastrophically unstable. The key difference here: replace only the activation function in the surface refinement MLP head (the `srf_head` in `train.py`) with `sin()`, leaving the main Transolver blocks untouched.

**Physical motivation:** The pressure distribution on an airfoil is fundamentally periodic/oscillatory as a function of arc-length position (pressure coefficient Cp oscillates from stagnation point around the suction surface to the pressure surface). The sinusoidal activation `sin(W*x + b)` naturally encodes this oscillatory structure, while ReLU/GELU cannot represent it without many neurons. The surface refinement MLP receives the slice attention features and maps them to pressure corrections — making this mapping sinusoidal aligns it with the known structure of the output.

**Why only the srf head (not full model):** SIREN requires careful initialization (specific weight initialization scheme with frequency scaling). In the full model, the instability came from SIREN conflicting with LayerNorm/Lion dynamics in the attention blocks. The srf_head is a 2-3 layer MLP applied post-attention — it's a leaf in the computation graph, so SIREN initialization can be applied cleanly without affecting upstream gradients.

**Complexity estimate:** ~8 lines. Replace `nn.GELU()` with `torch.sin` in the srf_head; apply SIREN initialization (W ~ Uniform(-sqrt(6/n_in), sqrt(6/n_in)) for first layer, x frequency factor for subsequent).

**Implementation sketch:**
```python
class SinActivation(nn.Module):
    def forward(self, x):
        return torch.sin(x)

# Replace in srf_head (the surface refinement MLP):
# Instead of nn.GELU(), use SinActivation()
# Apply SIREN init: nn.init.uniform_(layer.weight, -sqrt(6/fan_in), sqrt(6/fan_in))
# For hidden layers, multiply init range by omega_0 (default 30)
```

**Key hyperparameters:** `omega_0` in {1.0, 10.0, 30.0} — the frequency scaling parameter for SIREN. Start with omega_0=10.0 (lower than canonical 30.0 since we're not doing full coordinate mapping). Apply only to srf_head, not condition/slice layers.

**Expected impact:** p_in and p_tan — these are the metrics most dependent on sharp pressure peak representation. p_re (Reynolds-dependent smooth distributions) may not benefit.

**Key paper:** Sitzmann et al., "Implicit Neural Representations with Periodic Activation Functions", NeurIPS 2020. Section 3.1 describes the initialization scheme — critical to follow exactly.

**Confidence:** Moderate. The SIREN-in-head approach is novel and plausible. Risk is that Lion's sign-based updates conflict with the sinusoidal gradient landscape of the srf head — but since this is a small MLP, the impact is limited.

### 6. Contrastive Tandem-Single Representation Regularization

**Priority: MEDIUM. ~25 lines. Directly addresses OOD representation quality for tandem configs.**

Add an auxiliary contrastive loss that pushes apart the hidden representations of tandem samples from single-foil samples. The intuition: if tandem and single-foil samples are close together in hidden-state space (which the current model likely does since tandem configs are a minority of training data), the model is using the same representation for physically different flow structures. Contrastive separation forces the model to maintain distinct feature subspaces for each configuration type.

Specifically: for a batch containing both tandem and single-foil samples, compute the mean hidden state per configuration type (mean over nodes), then apply a margin-based contrastive loss that encourages `dist(tandem_mean, single_mean) > margin`.

**Why it might help:** The consistent gap between p_tan (29.1) and p_in (12.2) despite hundreds of training experiments suggests the model conflates tandem and single-foil representations. This is a structural regularization that directly discourages this conflation during training.

**Complexity estimate:** ~25 lines. Requires batch to contain both tandem and single-foil samples (already true with tandem_ramp).

**Implementation sketch:**
```python
# In training loop, after forward pass:
# model_out["hidden"]: [B, N, n_hidden]
# is_tandem: [B] bool mask from data loader

if is_tandem.any() and (~is_tandem).any():
    tandem_repr = model_out["hidden"][is_tandem].mean(dim=(0, 1))  # [n_hidden]
    single_repr = model_out["hidden"][~is_tandem].mean(dim=(0, 1))  # [n_hidden]
    cos_sim = F.cosine_similarity(tandem_repr.unsqueeze(0), single_repr.unsqueeze(0))
    contrastive_loss = F.relu(cos_sim - cfg.contrastive_margin)  # push below margin
    loss = main_loss + cfg.contrastive_weight * contrastive_loss
```

**Key hyperparameters:** `contrastive_margin` in {-0.3, -0.1, 0.0} (cosine similarity should be pushed below 0 = orthogonal); `contrastive_weight` in {0.01, 0.1}. Start with margin=-0.1 and weight=0.05.

**Expected impact:** p_tan primarily. Risk: if over-regularized, the shared features that ARE transferable between configurations get disrupted, hurting p_oodc.

**Key paper:** Chen et al., "A Simple Framework for Contrastive Learning of Visual Representations" (SimCLR), ICML 2020. The negative pair concept is adapted here to configuration-type pairs rather than augmentation pairs.

**Confidence:** Speculative but logically motivated. The direct OOD-representation interpretation of p_tan gap is plausible. The margin and weight are highly sensitive — recommend a 3-value sweep.

### 7. Progressive Surface Focus Schedule (Ramp surf_weight 1 to N)

**Priority: MEDIUM. ~8 lines. Directly targets surface MAE without changing architecture.**

The `surf_weight` hyperparameter (currently a fixed value) controls how much more the surface nodes are weighted relative to volume nodes in the loss. The hypothesis: starting with `surf_weight=1.0` (equal weighting) during the first 30-40 epochs allows the model to first learn a good global flow field, then ramping to the target `surf_weight` (e.g., 3.0 or 5.0) for the remaining epochs forces surface specialization once the volume representation is stable.

**Physical motivation:** This mirrors curriculum learning — easy before hard. The global flow field (volume nodes, smooth gradients) provides the physical context (wake structure, stagnation point location) that the model needs BEFORE it can accurately predict sharp surface features. Starting with equal weighting ensures these global features are learned first.

**Distinction from existing work:** The current training uses a fixed `surf_weight`. The `tandem_ramp` flag does something similar but for sample weighting, not node weighting. No experiment has tried dynamic surf_weight scheduling.

**Complexity estimate:** ~8 lines. Add a linear ramp in the training loop.

**Implementation sketch:**
```python
# In training loop, before loss computation:
ramp_epochs = cfg.surf_weight_ramp_epochs  # new flag, default 40
if epoch < ramp_epochs:
    effective_surf_weight = 1.0 + (cfg.surf_weight - 1.0) * (epoch / ramp_epochs)
else:
    effective_surf_weight = cfg.surf_weight
# Use effective_surf_weight instead of cfg.surf_weight in loss weighting
```

**Key hyperparameters:** `surf_weight_ramp_epochs` in {20, 40} — start with 40 (25% of total 160-epoch budget). Final `surf_weight` should match or exceed current best value.

**Expected impact:** p_in and p_tan (both surface pressure metrics). Smooth ramp may help the optimization avoid local minima where the model over-focuses on surface nodes before the volume representation is stable.

**Key paper:** Curriculum learning: Bengio et al., "Curriculum Learning", ICML 2009. The schedule concept is general; the specific application to loss weighting is novel here.

**Confidence:** Moderate. The curriculum motivation is sound. The specific ramp schedule needs tuning. Interaction with EMA (starts at epoch 140) is benign since EMA starts after ramp ends.

### 8. Coordinate Frame Normalization per-Foil (Tandem-Aware Input Encoding)

**Priority: MEDIUM. ~20 lines. Targets the representation of tandem geometry in input features.**

Currently, the input coordinates for tandem configurations are in a single global frame. The aft foil's coordinates are expressed relative to the global domain origin. This means the model must learn to handle the same NACA profile appearing at two different absolute positions in input space.

Proposed change: for tandem configurations, express the aft foil's coordinates in a LOCAL frame centered at the aft foil's leading edge, and add a new input feature indicating the offset of the aft foil's leading edge relative to the fore foil's leading edge. For single-foil configurations, the frame is unchanged.

**Why it might help:** The physics at the aft foil surface depends on the LOCAL pressure/velocity field around the aft foil, not the global position. By expressing the aft foil in a local frame, the model can reuse features learned for single-foil configurations when predicting aft-foil surface pressure. This is a form of equivariance that the current architecture lacks.

**Complexity estimate:** ~20 lines in model preprocessing (modifiable). Implemented as a model-level feature transformation in `train.py`.

**Implementation sketch (model-level, no data file modification):**
```python
# In Transolver.forward, before embedding:
# Detect tandem: condition encoding includes tandem flag or Re/AoA implies it
# For aft-foil nodes (boundary_id == 7):
#   Shift coordinates: x_aft_centered = x_aft - aft_leading_edge_coords
#   The aft leading edge position can be derived from the mesh as the min-x node of boundary_id=7
# Reconstruct x with locally-centered aft foil coordinates
```

**Key hyperparameters:** Whether to also normalize by chord length (divide by c=1.0 if all foils have unit chord). No additional hyperparameters.

**Expected impact:** p_tan primarily. No expected impact on p_in (single-foil, unchanged). Potential p_oodc improvement if OOD cases include novel gap/stagger configurations.

**Confidence:** Moderate. The equivariance argument is well-established in molecular property prediction (DimeNet, SE(3)-transformers) where local frame normalization is standard. The risk is that the `train.py` modification must correctly identify aft-foil nodes and compute the local frame without access to the data preparation pipeline.

### 9. Gradient Noise Injection (Langevin Dynamics Regularization)

**Priority: MEDIUM. ~10 lines. Distinct from SAM -- targets flat basin finding via different mechanism.**

SAM was tried (frieren #2086) and failed because it destabilizes Lion training — the second forward pass conflicts with Lion's sign-based update. The failure mechanism was: SAM perturbs weights by gradient direction, then Lion computes a different sign gradient from the perturbed weights, creating incoherent updates.

Gradient noise injection (Langevin dynamics) is fundamentally different: it adds small Gaussian noise DIRECTLY TO THE GRADIENTS after Lion computes them, before the weight update. This doesn't require a second forward pass and doesn't conflict with Lion's sign operation.

**Theoretical motivation:** Langevin dynamics (stochastic gradient Langevin dynamics, SGLD) adds noise proportional to `sqrt(2*eta/beta)` to gradient updates, which by the fluctuation-dissipation theorem causes the optimizer to sample from a Gibbs distribution over parameter space rather than converging to a point. In practice this means flat basins are favored (more probability mass = more samples) and sharp minima are penalized (Langevin particles escape sharp basins quickly).

**Why it might help:** The ensemble of 30 seeds (66-95) shows that seed diversity improves results — implying the model finds different local minima that are meaningfully different. Langevin noise during training encourages a SINGLE training run to explore more of the loss landscape, potentially finding a basin that is both lower and flatter.

**Complexity estimate:** ~10 lines. After `loss.backward()` and before `optimizer.step()`, add noise scaled by the learning rate.

**Implementation sketch:**
```python
# In training loop, after loss.backward():
if cfg.langevin_noise > 0.0:
    current_lr = scheduler.get_last_lr()[0]
    noise_scale = cfg.langevin_noise * math.sqrt(2.0 * current_lr)
    for param in model.parameters():
        if param.grad is not None:
            param.grad.data.add_(
                torch.randn_like(param.grad.data) * noise_scale
            )
# Then optimizer.step() as normal
```

**Key hyperparameters:** `langevin_noise` in {0.001, 0.01, 0.1} — start with 0.01. Scale with LR so noise anneals with the learning rate (cosine schedule). Disable during EMA collection phase (epoch >= 140).

**Expected impact:** Broad improvement across all metrics if it finds a flatter basin. Primarily expected to improve ensemble-like variance without requiring multiple seeds.

**Key paper:** Welling & Teh, "Bayesian Learning via Stochastic Gradient Langevin Dynamics", ICML 2011. The modern connection to flat-basin finding: Chaudhari et al., "Entropy-SGD: Biasing Gradient Descent Into Wide Valleys", ICLR 2017.

**Confidence:** Moderate. Langevin noise is well-studied theoretically. The interaction with Lion (sign-based) is uncertain — Lion discards gradient magnitude so Langevin noise only affects direction. This may make it ineffective. Start with high noise to test if signal is detectable.

### 10. Fourier Feature Augmentation for Mesh Coordinates

**Priority: MEDIUM. ~15 lines. Direct upgrade to input encoding, targets coordinate representation quality.**

The input coordinate features (x, y positions) are currently encoded linearly before the embedding layer. Fourier Feature Networks (Tancik et al., NeurIPS 2020) showed that mapping coordinates through random Fourier features before the MLP dramatically improves the ability to represent high-frequency functions — exactly the kind of sharp pressure gradients at leading edges.

**Why it might help:** The spectral norm decay of neural networks without Fourier features effectively low-pass filters the spatial representation. For airfoil pressure distributions, the suction peak at the leading edge is a high-frequency spatial feature (~1-2 node widths wide). Fourier feature encoding explicitly provides the model with periodic basis functions at the relevant spatial frequencies.

**Distinct from LinearNO (dead end #2033-2038):** LinearNO replaced the attention mechanism with a linear spectral operator (fundamentally different). Fourier feature encoding is a simple preprocessing step on the input coordinates only — the attention mechanism is unchanged.

**Complexity estimate:** ~15 lines. Add a `FourierFeatures` layer before the input embedding.

**Implementation sketch:**
```python
class FourierFeatures(nn.Module):
    def __init__(self, input_dim=2, n_freq=16, sigma=1.0):
        super().__init__()
        B = torch.randn(input_dim, n_freq) * sigma
        self.register_buffer('B', B)  # fixed random projections
    
    def forward(self, x_coords):  # [B, N, 2] xy coordinates
        proj = 2 * math.pi * x_coords @ self.B  # [B, N, n_freq]
        return torch.cat([torch.sin(proj), torch.cos(proj)], dim=-1)  # [B, N, 2*n_freq]

# In Transolver.__init__:
self.fourier = FourierFeatures(input_dim=2, n_freq=16, sigma=cfg.fourier_sigma)
# Concatenate Fourier features with the other input features before embedding
```

**Key hyperparameters:** `n_freq` in {8, 16, 32} — start with 16 (adds 32 features); `fourier_sigma` in {0.1, 1.0, 10.0} — controls frequency range; lower sigma = lower frequencies. For airfoil coordinates normalized to [-1, 1], sigma=1.0 is a reasonable starting point. Apply only to spatial coordinates (first 2 dims of input x), not the flow condition features.

**Expected impact:** p_in and p_tan — high-frequency spatial features. Potentially large improvement if the current coordinate encoding is the bottleneck for suction peak representation.

**Key paper:** Tancik et al., "Fourier Features Let Networks Learn High Frequency Functions in Low Dimensional Domains", NeurIPS 2020. Section 4 shows 10-100x improvement in frequency representation for 2D coordinate regression tasks — directly applicable to airfoil surface pressure prediction.

**Confidence:** Moderate-strong. The Fourier feature result is well-validated in neural radiance fields (NeRF), implicit neural representations, and mesh-based simulations. The transfer to our Transolver architecture is conceptually clean. Main risk: the current embedding layer may already implicitly learn Fourier-like features; if so, the gain is marginal. Test with sigma=1.0 first.
