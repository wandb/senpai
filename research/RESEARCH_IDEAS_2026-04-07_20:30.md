## Idea 1: Decoupled Tandem Slice Projection

**Slug:** `decouple-tandem-slice`

**What it is:** Enable `--decouple_slice`, a flag already implemented in `Physics_Attention_Irregular_Mesh` (line 166 of train.py). When active, tandem samples use a SEPARATE orthogonally-initialized slice projection matrix (`in_project_slice_tandem`) while single-foil samples continue to use `in_project_slice`. This means the physics slice routing is learned independently for tandem vs. single-foil configurations.

**Why this hasn't been tried:**
- `decouple_slice` appears in the codebase and was mentioned in PR descriptions (e.g. PR #2155 results: "worth testing if not already done") but was never run as a standalone experiment on any Phase 6 baseline.
- It does NOT appear in any merged PR's command arguments.

**Mechanism:** In Transolver, slice routing assigns each mesh node to a mixture of physics slices. For tandem samples, two interacting foils create flow topologies fundamentally different from single-foil cases: inter-foil pressure channels, wake interactions, blockage effects. The current single shared `in_project_slice` matrix must simultaneously encode both single-foil and tandem routing patterns. `decouple_slice` gives tandem samples their own learned routing vocabulary — allowing the model to specialize slices for tandem-specific physics (e.g., dedicated slices for inter-foil channel nodes, wake region, aft-foil leading edge) without contaminating the single-foil routing.

**Why it should help p_tan:** The tandem transfer split evaluates on NACA6416 as front foil — an OOD geometry. With decoupled routing, the tandem projection learns to route by functional physics region rather than shape-specific features, improving generalization to unseen tandem geometries.

**Zero-risk initialization:** `in_project_slice_tandem` is orthogonally initialized (same as `in_project_slice`) — at epoch 0, tandem routing starts from a reasonable basis. The model can learn to diverge or stay similar depending on what the data supports.

**Parameter overhead:** 96 × (n_hidden // n_head) = 96 × 48 = 4,608 floats (18 KB). Negligible.

**VRAM:** Zero additional VRAM (same computation graph, just separate weight matrix for tandem).

**Expected impact:** -2 to -5% p_tan. Modest p_oodc improvement possible. Low risk of regression since initialization is safe.

**Confidence:** MEDIUM-HIGH. The decouple_slice flag exists precisely for this purpose and was never tested on the current production baseline. It's the cleanest untested hyperparameter in the codebase.

**Implementation:** Just add `--decouple_slice` to the baseline command. Zero code changes needed.

## Idea 2: Flow-Regime Conditioned SRF via FiLM on AoA and Umag

**Slug:** `srf-flow-film`

**What it is:** Add FiLM (Feature-wise Linear Modulation) conditioning on global flow regime (AoA, Umag) to the main `SurfaceRefinementHead`. The SRF head currently corrects surface predictions using only local hidden features and base predictions — it has no knowledge of what flow regime it is operating in. A small 2-layer MLP takes (normalized_AoA, normalized_Umag) as input and outputs (scale, shift) vectors applied after the first hidden layer of the SRF MLP.

**Why this is distinct from prior FiLM experiments:**
- Phase 2/3 FiLM (PRs #1103, #1461): conditioned the BACKBONE (TransolverBlock) on Re+AoA. The backbone already receives global flow information through input features. This targets the SRF HEAD specifically.
- AftFoilRefinementHead with `--film` (merged, uses gap/stagger conditioning): conditions on tandem GEOMETRY (gap=x[0,22], stagger=x[0,23]). This conditions on FLOW REGIME (AoA, Umag). These are orthogonal axes.
- `--adaln_output` (merged): conditions the last TransolverBlock's output on (Umag, AoA). This is backbone conditioning. The SRF head, which post-processes the backbone output, receives no flow conditioning.

**Mechanism:** The correction needed for a surface node at extreme AoA (separated flow, very different pressure gradients) is fundamentally different from the correction at cruise AoA. Similarly, the SRF correction at high Umag differs from low Umag in the p_oodc split. By giving the SRF an explicit flow-regime signal, it can learn to apply different corrections for different operating points — a key requirement for OOD generalization.

**Physical motivation:** The surface pressure coefficient Cp = (p - p_inf) / q scales with flow conditions in ways that the Transolver backbone may not fully capture. The SRF correction for a given surface location in attached flow (cruise AoA) is a small residual; in separated flow (extreme AoA), it is a large correction to capture suction peak collapse. FiLM conditioning allows the SRF to modulate its correction magnitude and direction based on flow regime without needing to infer this from local features alone.

**Expected improvement:** -3 to -6% p_oodc (extreme AoA/Umag is the primary p_oodc driver), -2 to -4% p_re (OOD Re changes the local pressure field systematically). p_tan likely flat.

**Confidence:** MEDIUM. The physical motivation is clear. The risk is that the backbone already provides regime information through the hidden state, making the FiLM redundant. But the zero-init safety (FiLM outputs initialized to identity: scale=1, shift=0) means no regression risk.

**Implementation notes:**
- Add config flags: `srf_flow_film: bool = False`
- In `SurfaceRefinementHead.__init__`, add when `flow_film=True`:
  ```python
  self.flow_film_net = nn.Sequential(
      nn.Linear(2, 128), nn.SiLU(),
      nn.Linear(128, hidden_dim * 2)
  )
  nn.init.zeros_(self.flow_film_net[-1].weight)
  nn.init.zeros_(self.flow_film_net[-1].bias)
  ```
- In `SurfaceRefinementHead.forward`, accept optional `flow_cond: [B, 2]` (normalized AoA, Umag), expand to surface nodes, apply after first hidden activation: `x = x * (1 + scale) + shift`
- At call site, pass `flow_cond` from the per-sample `condition` tensor already available in the training loop (it's used by `--adaln_output`)
- The `condition` tensor in the training loop (from `data/prepare_multi.py`) carries (Umag, AoA) — these are already normalized. Use them directly.

## Idea 3: Two-Pass Iterative SRF — Sequential Surface Refinement

**Slug:** `srf-two-pass`

**What it is:** Run the SurfaceRefinementHead twice in sequence. Pass 1 produces `correction1 = SRF(hidden, base_pred)`. Pass 2 produces `correction2 = SRF2(hidden, (base_pred + correction1).detach())`. Final prediction: `pred = base_pred + correction1 + correction2`. The two passes have SEPARATE weight matrices (SRF1 and SRF2 are independent modules). The `.detach()` prevents gradient from flowing back through SRF1 via SRF2's loss, keeping each pass as an independent corrector.

**Why this hasn't been tried:**
- PR #1882 (Iterative Residual Refinement, Phase 5): trained a 2-STAGE full model with separate training phases. This is fundamentally different — it's 2 stages of the ENTIRE training pipeline with frozen first stage.
- PR #2103 (Iterative Weight-Tied Transolver, Phase 6): reused Transolver BACKBONE blocks iteratively. Closed as failed.
- The current SRF head (PR #1935, merged) applies a single-pass correction with no iteration. No experiment has tested iterating the SRF head itself.

**Mechanism:** A single linear-then-nonlinear pass through the SRF MLP can learn one "mode" of correction. For pressure, there are multiple independent error sources: (1) slow global shifts from backbone bias at a given Re; (2) local sharp features at leading/trailing edge; (3) inter-foil coupling errors in tandem. These require different spatial frequencies of correction. A second pass, operating on the already-corrected prediction, can learn to correct residuals that the first pass systematically misses — the residuals of the residuals.

This is the same principle as boosting in ensemble learning: each weak corrector (a SRF pass) focuses on the errors left by the previous corrector. XGBoost and gradient boosting achieve state-of-the-art on tabular data through exactly this sequential residual approach.

**SRF2 can be smaller:** Since SRF1 already handles the main correction, SRF2 only needs to learn the second-order residual. Use smaller hidden dim for SRF2: `--surface_refine2_hidden 96 --surface_refine2_layers 2`.

**Zero-init safety:** SRF2's output layer is zero-initialized — at epoch 0, correction2=0 and the model is identical to the single-pass SRF baseline.

**Parameter overhead:** SRF2 at hidden=96, 2 layers: (192+3)×96 + 96×96 + 96×3 ≈ 28,000 params. Negligible. VRAM: same compute graph structure, ~+30 MB.

**Expected impact:** -2 to -4% p_tan (multi-pass correction of tandem surface geometry which is harder to capture in one pass), -2 to -3% p_oodc. The second pass particularly targets the leading-edge suction peak error, which is a second-order residual from the first SRF pass.

**Confidence:** MEDIUM. Sequential boosting of residuals is a well-established ML principle. The zero-init safety is high. The main risk is that the second pass overfits to training-set residuals without improving OOD residuals — but the detach prevents gradient interference and the SRF2 parameters are small enough that overfitting risk is low.

**Implementation notes:**
- Add config flags: `surface_refine2: bool = False`, `surface_refine2_hidden: int = 96`, `surface_refine2_layers: int = 2`
- Instantiate a second `SurfaceRefinementHead(n_hidden=cfg.n_hidden, out_dim=3, hidden_dim=96, n_layers=2)` as `refine_head2` alongside the existing `refine_head`
- In the training/eval forward pass, after computing `pred_surf = base_pred_surf + refine_head(hidden_surf, base_pred_surf)`, add:
  ```python
  if cfg.surface_refine2:
      correction2 = refine_head2(hidden_surf, pred_surf.detach())
      pred_surf = pred_surf + correction2
  ```
- Save `refine_head2.pt` alongside `refine_head.pt` in the checkpoint logic
- Apply to both the main SRF and aft-foil SRF (or just the main SRF as a first test)
