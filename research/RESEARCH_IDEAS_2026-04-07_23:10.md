## Idea 1: Global Nyström Pathway in Slice Attention (GFocal-style)

**Slug:** `global-nystrom-attn`

**Mechanism:**
The current Physics_Attention_Irregular_Mesh uses only local "slice" attention — nodes are
assigned to 96 slices and cross-slice attention is performed on slice tokens. This is excellent
for local routing but cannot model truly long-range correlations that cross slice boundaries.
The GFocal paper (arXiv 2508.04463) shows that adding a global Nyström attention pathway
(using ~16 landmark tokens via random selection or k-means) in parallel to the local focal
pathway dramatically improves operator accuracy on irregular meshes, with a lightweight
convolution gate controlling the blend.

For our CFD problem, global interactions matter in two specific ways: (1) the leading-edge
pressure coefficient integrates contributions from all foil surface nodes simultaneously
(Kutta condition coupling), and (2) tandem-foil wake interference requires nodes near the aft
foil to "see" the fore foil's trailing edge — a global relationship the 96 slices may not
span consistently.

**Implementation:**
In `Physics_Attention_Irregular_Mesh.forward`, after computing `attn_output` (the existing
slice pathway output), add a parallel Nyström branch:

```python
# Nyström global pathway (new flag: --global_nystrom_attn, landmarks=16)
if self.global_nystrom:
    # Select 16 landmark tokens via uniform striding
    n_land = 16
    stride = max(1, N // n_land)
    land_idx = torch.arange(0, N, stride, device=fx.device)[:n_land]
    landmarks = fx[:, land_idx, :]  # [B, 16, H]
    # Standard scaled dot-product attention: all nodes attend to landmarks
    q_g = self.nystrom_q(fx)       # [B, N, H]
    k_g = self.nystrom_k(landmarks) # [B, 16, H]
    v_g = self.nystrom_v(landmarks) # [B, 16, H]
    attn_g = (q_g @ k_g.transpose(-2,-1)) / math.sqrt(H)
    attn_g = attn_g.softmax(dim=-1)  # [B, N, 16]
    global_out = attn_g @ v_g        # [B, N, H]
    # Learned gate to blend global and local outputs
    gate = torch.sigmoid(self.nystrom_gate(fx))  # [B, N, H], init weight=0
    attn_output = attn_output + gate * global_out
```

New parameters: `nystrom_q/k/v` are Linear(n_hidden, n_hidden) with orthogonal init;
`nystrom_gate` is Linear(n_hidden, n_hidden) with zero init (starts fully closed, so initial
behavior is unchanged).

New flag: `--global_nystrom_attn` (bool). Apply to all 3 TransolverBlocks.

n_hidden=192, n_land=16 adds ~3*192^2 ≈ 110K params — acceptable.

**Training command:**
```bash
python train.py --global_nystrom_attn --wandb_group global-nystrom-attn
```
(All other hyperparameters remain at current defaults.)

**Expected impact:** Moderate-high. Should primarily help p_tan (tandem wake coupling) and
p_oodc (cross-domain generalization). The gate initialization at zero means no regression risk.
Similar mechanisms in GFocal improve accuracy 8-15% on PDE benchmarks over slice-only attention.

**Confidence:** Medium. GFocal was validated on Darcy flow and NS equations; the CFD mesh
structure is similar but the specific benefit on pressure prediction is unproven here.

## Idea 2: RANS Residual Physics Consistency Loss

**Slug:** `rans-consistency-loss`

**Mechanism:**
The current model predicts (Ux, Uy, p) but never enforces that these predictions are
consistent with the underlying RANS equations. AeroDiT (arXiv 2412.17394) demonstrated that
adding mass conservation (∇·u ≈ 0) and simplified momentum residual as auxiliary losses
during training acts as a powerful physics regularizer that is especially valuable for OOD
generalization — the model is forced to occupy the manifold of physically realizable solutions.

For 2D incompressible flow, mass conservation simplifies to: ∂Ux/∂x + ∂Uy/∂y = 0. On our
irregular mesh this can be approximated via finite differences using the known node coordinates.
For adjacent node pairs (node i, node j with dx=x_j-x_i, dy=y_j-y_i):
  divergence ≈ (Ux_j - Ux_i)/dx + (Uy_j - Uy_i)/dy

This is approximated via a precomputed neighbor graph (k=4 nearest volume nodes) and does not
require any graph convolution — it is a simple auxiliary loss on the predictions.

**Implementation:**
In the training loop, after computing base predictions for volume nodes:

```python
if cfg.rans_consistency_loss and not is_surface_node:
    # For each volume node, find k=4 nearest neighbors (precomputed per sample)
    # Approximate divergence via finite differences on predicted velocities
    # neighbor_dx, neighbor_dy: [N_vol, k] precomputed from coords
    ux_pred_vol = preds[vol_mask, 0]  # [N_vol]
    uy_pred_vol = preds[vol_mask, 1]  # [N_vol]
    # Neighbor differences
    ux_j = ux_pred_vol[neighbor_idx]  # [N_vol, k]
    uy_j = uy_pred_vol[neighbor_idx]
    div_approx = ((ux_j - ux_pred_vol.unsqueeze(1)) / (neighbor_dx + 1e-6) +
                  (uy_j - uy_pred_vol.unsqueeze(1)) / (neighbor_dy + 1e-6))
    loss_div = div_approx.pow(2).mean()
    loss = loss + cfg.rans_weight * loss_div  # rans_weight=0.01
```

Neighbor indices and dx/dy are precomputed once per batch from the `pos` tensor.
New flag: `--rans_consistency_loss` with `--rans_weight 0.01`.

**Key implementation detail:** Only apply to volume nodes (not surface), since surface pressure
boundary conditions make surface divergence ill-posed. PCGrad compatibility: treat the RANS
loss as a 4th task in PCGrad gradient surgery — add `rans_grad` to the gradient conflict
resolution.

**Training command:**
```bash
python train.py --rans_consistency_loss --rans_weight 0.01 --wandb_group rans-consistency
```

**Expected impact:** Moderate. Primary benefit should be p_re (Reynolds number OOD) and p_oodc
(geometry OOD) — exactly the cases where physical consistency matters most. AeroDiT reports
15-20% improvement on velocity divergence error; pressure improvement is secondary.

**Confidence:** Medium-low. The neighbor graph approximation introduces discretization error.
The benefit depends on whether the model's failure modes are physically inconsistent predictions
(likely for OOD) vs simply insufficient capacity (less likely given current architecture depth).

## Idea 3: Attention Logit Noise — Targeted Backbone Perturbation

**Slug:** `attn-logit-noise`

**Mechanism:**
PR #2254 (Backbone Hidden Noise) failed because additive noise on hidden states compounds
through residual connections. The autopsy identified "attention-level" noise as the mechanistically
correct target. The slice attention logits — the `attn_logits` tensor computed just before
the slice-token softmax — are the ideal injection point: perturbation there affects slice
routing stochasticity without propagating through residuals.

This is directly analogous to dropout in attention logits (Zehui Lin et al., "DropAttention",
2019), which has been shown to act as a strong regularizer for OOD generalization specifically
by preventing over-reliance on specific attention patterns. For our model, where tandem OOD
failures stem from the model routing aft-foil wake nodes to "wrong" slices, logit noise forces
the model to learn attention patterns that are robust to moderate routing perturbations.

**Implementation:**
In `Physics_Attention_Irregular_Mesh.forward`, after computing `attn_logits`:

```python
# Attention logit noise (new flag: --attn_logit_noise_std, default=0.0)
# attn_logits shape: [B, n_head, slice_num, slice_num]
if self.training and attn_logit_noise_std > 0:
    noise = torch.randn_like(attn_logits) * attn_logit_noise_std
    attn_logits = attn_logits + noise
```

Critically: only apply during training (not eval/EMA), and only to the `attn_logits` (cross-
slice cosine similarity product before softmax), NOT to `slice_weights` (the per-node slice
assignment logits). The slice assignment is a separate routing decision that should remain
deterministic.

New flag: `--attn_logit_noise_std` with suggested value `0.05`. Apply to all TransolverBlocks.

**Training command:**
```bash
python train.py --attn_logit_noise_std 0.05 --wandb_group attn-logit-noise
```

**Expected impact:** Moderate. Most likely to help p_tan and p_oodc — the OOD metrics where
attention routing rigidity is the hypothesized failure mode. p_in and p_re may be unaffected
or slightly worse. Risk of over-regularization is low at σ=0.05 since logits are O(1) scale.

**Confidence:** Medium. DropAttention literature validates the mechanism, but the CFD mesh's
non-uniform node density means logit noise affects physically sparse regions disproportionately.
Worth checking if tandem-specific filtering helps (only apply when `is_tandem=True`).

## Idea 4: Foil Role Embedding — Explicit Fore/Aft Identity for Surface Nodes

**Slug:** `foil-role-embedding`

**Mechanism:**
In the current model, the distinction between fore-foil and aft-foil surface nodes is encoded
implicitly via the TE coordinate frame features (dx_fore, dy_fore, r_fore, dx_aft, dy_aft,
r_aft) and the gap/stagger features. However, there is no explicit learned embedding that
tells the model "this node is on the aft foil" — the model must infer this from the coordinate
geometry.

For tandem aerodynamics, the fore and aft foils play fundamentally different aerodynamic roles:
the fore foil generates the wake that impinges on the aft foil, while the aft foil operates in
an asymmetric inflow field. This asymmetry is physically meaningful and should be a first-class
input, not something the model must discover from coordinate geometry.

The stagger sign (positive = fore foil is upstream in x, negative = aft foil is upstream) also
flips the aerodynamic "role" of each foil. A 2-dim learned embedding (fore_embed, aft_embed)
injected per surface node — and zeroed for single-foil samples — gives the model direct
structural information about aerodynamic role.

**Implementation:**
In `Transolver.forward`, after `self.preprocess(x)`:

```python
if cfg.foil_role_embedding:
    # fore_embed: [1, n_hidden], aft_embed: [1, n_hidden] — learned embeddings
    # is_fore_foil: [B, N] boolean — node belongs to fore foil (boundary_id in {5,6})
    # is_aft_foil: [B, N] boolean — node belongs to aft foil (boundary_id == 7)
    # For single-foil samples (gap==0), both masks are False
    is_tandem_b = x[:, 0, 21].abs() > 0.01  # [B]
    role_emb = torch.zeros_like(fx)  # [B, N, n_hidden]
    for b in range(B):
        if is_tandem_b[b]:
            fore_mask = is_fore_foil[b]  # precomputed from boundary_id
            aft_mask = is_aft_foil[b]
            role_emb[b, fore_mask] = self.fore_embed  # broadcast [n_hidden]
            role_emb[b, aft_mask] = self.aft_embed
    fx = fx + role_emb
```

`self.fore_embed` and `self.aft_embed` are `nn.Parameter(torch.zeros(n_hidden))` — zero
initialized so initial behavior is unchanged.

The boundary_id is already available in the data (used by AftFoilRefinementHead for ID=7
masking). Fore foil = boundary IDs {5, 6}, aft foil = boundary ID {7}.

New flag: `--foil_role_embedding`.

**Training command:**
```bash
python train.py --foil_role_embedding --wandb_group foil-role-embedding
```

**Expected impact:** Moderate. Should primarily improve p_tan. The zero initialization means
no regression risk on p_in/p_re. The mechanism addresses a genuine gap in the input
representation — the model currently has no direct "I am the wake-receiving foil" signal.

**Confidence:** Medium-high. The mechanism is simple and well-motivated. The zero init means
it can only help, never catastrophically fail. The main risk is that the coordinate frame
features already encode sufficient foil identity implicitly.

## Idea 5: Hard Sample Replay Buffer (PER-Style, Sampling Frequency)

**Slug:** `hard-sample-replay`

**Mechanism:**
PR #2257 (tanjiro, in-flight) uses focal reweighting — upweighting hard samples by loss^0.5
within a minibatch. The replay buffer approach is orthogonal: instead of changing gradient
magnitudes, it changes *how often* hard samples appear in training. This has a fundamentally
different computational graph and interacts differently with PCGrad.

Inspired by Prioritized Experience Replay (Schaul et al. 2015, NeurIPS) and curriculum-by-loss
methods in NLP fine-tuning, the idea is:

1. Maintain a running exponential moving average of per-sample losses (updated each time a
   sample appears in a batch).
2. After epoch 30 (once the model has seen each sample ~3 times), bias sampling toward
   high-EMA-loss samples using a priority distribution P(i) ∝ loss_ema[i]^α, α=0.6.
3. Implement via `WeightedRandomSampler` — no changes to loss computation, fully compatible
   with PCGrad.

This is distinct from focal reweighting (#2257) because: (1) it changes sampling frequency
not gradient magnitude, (2) it compounds: a hard sample gets seen more often AND each time
it updates the EMA, making the priority adaptive, (3) no interaction with PCGrad gradient
surgery.

**Implementation:**
```python
# In Dataset/DataLoader setup:
class PrioritizedSampler(torch.utils.data.Sampler):
    def __init__(self, n_samples, alpha=0.6):
        self.priorities = torch.ones(n_samples)  # uniform initially
        self.alpha = alpha

    def update(self, indices, losses):
        # Call after each batch with per-sample losses
        for idx, loss in zip(indices, losses):
            self.priorities[idx] = 0.9 * self.priorities[idx] + 0.1 * loss.item()

    def __iter__(self):
        probs = self.priorities ** self.alpha
        probs = probs / probs.sum()
        return iter(torch.multinomial(probs, len(self.priorities), replacement=True).tolist())
```

New flag: `--hard_sample_replay` with `--replay_alpha 0.6` and `--replay_start_epoch 30`.
The sampler is initialized uniform and transitions to priority-based at epoch 30.

**Training command:**
```bash
python train.py --hard_sample_replay --replay_alpha 0.6 --replay_start_epoch 30 \
    --wandb_group hard-sample-replay
```

**Expected impact:** Moderate. Should help p_tan most — tandem samples are consistently harder
and will accumulate high EMA loss, increasing their sampling frequency. p_oodc may also benefit.
Risk: tandem over-sampling could hurt p_in by reducing single-foil gradient signal.

**Confidence:** Medium. PER is well-validated in RL and curriculum learning literature. The
main unknown is whether tandem samples' difficulty is due to insufficient exposure (replay
helps) or insufficient model capacity for tandem physics (replay does not help). Check whether
tandem p_tan is consistently worse across epochs — if yes, replay is the right intervention.

## Idea 6: Dual EMA Checkpoint Selection — Separate Slow/Fast Models per Validation Split

**Slug:** `dual-ema-checkpoint`

**Mechanism:**
The current EMA uses a single decay=0.998 model for all 4 validation metrics. But the optimal
checkpoint for p_tan (which needs longer training integration to see enough tandem samples)
may differ from the optimal checkpoint for p_re (which benefits from aggressive early learning
before the model memorizes in-distribution patterns). The 4-split checkpoint selection (PR
#1122) selects the single best checkpoint across all 4 metrics — but with a single EMA model,
this is a compromise.

Dual EMA maintains two shadow models simultaneously:
- `ema_fast`: decay=0.999 — tracks recent weights closely (fast adaptation)
- `ema_slow`: decay=0.996 — heavily smoothed long-run average (stable generalization)

At inference time, use `ema_fast` for p_in and p_re (in-distribution metrics that benefit
from recent convergence) and `ema_slow` for p_tan and p_oodc (OOD metrics that benefit from
longer averaging). The checkpoint selection picks the best configuration of (fast, slow) across
all epochs.

This is inspired by the "model soups" and diverse checkpoint ensembles literature, but
implemented as a training-time mechanism with zero inference overhead — one of the two EMA
models is always used per metric.

**Implementation:**
```python
# Maintain two EMA models
ema_fast = EMA(model, decay=0.999)
ema_slow = EMA(model, decay=0.996)

# After each optimizer step:
ema_fast.update()
ema_slow.update()

# Validation: run each EMA on its assigned splits
val_in_loss = validate(ema_fast, val_in_loader)
val_oodc_loss = validate(ema_slow, val_oodc_loader)
val_tan_loss = validate(ema_slow, val_tan_loader)
val_re_loss = validate(ema_fast, val_re_loader)
# Save both EMA states; checkpoint when composite metric improves
```

New flag: `--dual_ema` with `--ema_fast_decay 0.999` and `--ema_slow_decay 0.996`.

**Training command:**
```bash
python train.py --dual_ema --ema_fast_decay 0.999 --ema_slow_decay 0.996 \
    --wandb_group dual-ema-checkpoint
```

**Expected impact:** Moderate. The current p_oodc=7.643 and p_tan=28.341 are the hardest
metrics. Using a slower EMA for OOD splits is motivated by the observation that single-model
p_tan already beats the 16-seed ensemble, suggesting the model does learn tandem physics but
checkpoint selection may not be optimal. Minimal regression risk on p_in/p_re since ema_fast
decay=0.999 matches the current single EMA.

**Confidence:** Medium. The hypothesis that different EMA decays are optimal for different
splits is plausible but not yet validated in this setting. The mechanism is extremely low-risk
(no architectural changes) and cheap to implement.

## Idea 7: SRF Arc-Length Positional Encoding

**Slug:** `srf-arc-length-pe`

**Mechanism:**
The SurfaceRefinementHead currently takes as input `[hidden (192-dim), base_pred (3-dim)]`
for each surface node. It has no information about *where on the airfoil surface* each node
is located — leading edge vs trailing edge vs pressure-side midchord vs suction-side midchord
all receive the same function form in the SRF.

This is categorically different from PR #2223 (Surface Arc-Length PE, which added positional
encoding to the backbone input features). Here we are adding arc-length position as an
additional input *only to the SRF head*, after the backbone has already processed the node.
The SRF's job is precisely localized surface correction, and it needs to know the geometric
context to apply the right correction profile.

Arc-length position s ∈ [0, 1] along the airfoil surface (0 = trailing edge, 0.5 = leading
edge, 1 = trailing edge again going the other way) can be computed from the node coordinates
and the known surface ordering. For tandem foils, we need s per foil (s_fore ∈ [0,1],
s_aft ∈ [0,1]).

Encode s using 8-dim sinusoidal PE: `[sin(2πks), cos(2πks)]` for k=1,2,3,4.

**Implementation:**
In `SurfaceRefinementHead.__init__`, change `in_dim`:
```python
in_dim = n_hidden + out_dim + 8  # +8 for arc-length PE
```

In the training loop, before calling `srf_head`:
```python
# Compute arc-length PE for surface nodes
s = compute_arc_length_position(surf_coords)  # [M_s] in [0,1]
k = torch.arange(1, 5, device=s.device).float()
arc_pe = torch.cat([torch.sin(2*math.pi*k*s.unsqueeze(1)),
                    torch.cos(2*math.pi*k*s.unsqueeze(1))], dim=-1)  # [M_s, 8]
# Concatenate to SRF input
srf_input = torch.cat([surf_hidden, base_pred_surf, arc_pe], dim=-1)
correction = srf_head(srf_input)
```

`compute_arc_length_position` uses cumulative chord-length parameterization from sorted
surface coordinates (already available from boundary_id masking).

New flag: `--srf_arc_length_pe`.

**Training command:**
```bash
python train.py --srf_arc_length_pe --wandb_group srf-arc-length-pe
```

**Expected impact:** Moderate. The SRF is the final prediction stage and adding position
context to it is mechanistically motivated — pressure corrections at the leading edge are
qualitatively different from those at the trailing edge (suction peak vs pressure recovery).
This is especially relevant for p_tan where the aft foil's pressure distribution is highly
non-uniform due to wake impingement.

**Confidence:** Medium-high. The mechanism is clear and the implementation is isolated to the
SRF head (no backbone changes). Low regression risk. The main uncertainty is whether the
backbone hidden state already encodes sufficient positional information for the SRF to infer
location implicitly — the TE coordinate frame features may have already saturated this.

## Idea 8: Late Stochastic Depth (Epoch 80+)

**Slug:** `late-stochastic-depth`

**Mechanism:**
Stochastic Depth (Huang et al. 2016) randomly drops entire residual blocks during training,
forcing other layers to develop redundant representations. In the vision transformer literature
(DeiT, ConvNeXt), this is a strong regularizer applied from epoch 0. For our model, prior
experiments applying stochastic depth from epoch 0 failed — the model failed to converge.

The crucial insight is the timing. From epoch 0 to ~70, the model is still learning the basic
flow physics — dropping layers destroys the learning signal. After epoch 80, the model has
converged to a reasonable solution and stochastic depth acts as a targeted OOD regularizer
rather than a convergence obstacle.

Late stochastic depth (survival_prob=0.90 for block 1, 0.95 for block 2; block 0 is never
dropped) forces the model to learn predictions that are robust to missing intermediate
representations. This should improve p_oodc and p_tan where the model may be over-relying on
block-specific features learned from training distribution geometry.

**Implementation:**
In `TransolverBlock.forward`:
```python
# Late stochastic depth (new flag: --late_stochastic_depth)
# survival_prob: [0.90, 0.95, 1.0] for blocks [0, 1, 2] respectively
# Only active after epoch >= 80 (pass epoch counter as training state)
if self.training and self.stochastic_depth_prob < 1.0 and epoch >= 80:
    if torch.rand(1).item() > self.stochastic_depth_prob:
        return fx  # skip this block entirely (identity)
```

`stochastic_depth_prob` is set per block: block 0 gets 0.90, block 1 gets 0.95, block 2
(last) gets 1.0 (never dropped, since it contains the output head).

New flags: `--late_stochastic_depth` with `--sd_start_epoch 80` and `--sd_prob_0 0.90` and
`--sd_prob_1 0.95`.

Note: The epoch counter must be passed into the block forward call or stored as module state
(e.g., `block.current_epoch = epoch` before the forward pass).

**Training command:**
```bash
python train.py --late_stochastic_depth --sd_start_epoch 80 --sd_prob_0 0.90 --sd_prob_1 0.95 \
    --wandb_group late-stochastic-depth
```

**Expected impact:** Moderate. The timing hypothesis (apply only after epoch 80, not from
epoch 0) is the key bet. If the prior failures were purely due to early-epoch convergence
interference, this directly addresses that failure mode. Expected benefit on p_oodc and p_re
(OOD generalization). Low-to-moderate risk: the survival probabilities (0.90, 0.95) are
conservative enough that training should remain stable.

**Confidence:** Medium. The late-application timing is well-motivated by the observed failure
mode of early stochastic depth, but this specific timing hypothesis has not been tested in
this codebase. Comparable techniques in NLP (curriculum masking scheduled to appear after
initial convergence) validate the general principle.

## Priority Ranking

1. **Foil Role Embedding** (Idea 4) — zero-risk init, directly addresses a genuine gap in
   tandem representation, low complexity. Highest probability of improving p_tan.

2. **SRF Arc-Length PE** (Idea 7) — isolated to SRF head, mechanistically clear, orthogonal
   to all in-flight experiments. Good candidate for p_tan improvement.

3. **Attention Logit Noise** (Idea 3) — direct follow-up from PR #2254 autopsy, targets
   the specific failure mode identified. Mechanistically sound.

4. **Dual EMA Checkpoint** (Idea 6) — zero architectural risk, could improve all 4 metrics
   through better checkpoint selection. Very cheap to implement.

5. **Hard Sample Replay** (Idea 5) — orthogonal to #2257 (focal reweighting, in-flight),
   complements it by changing sampling frequency instead of gradient magnitude.

6. **Late Stochastic Depth** (Idea 8) — higher risk than #3-5 due to training stability
   concerns, but directly motivated by prior failure analysis.

7. **Global Nyström Pathway** (Idea 1) — largest architectural change, good theory, but
   higher implementation complexity and unknown CFD-specific behavior.

8. **RANS Consistency Loss** (Idea 2) — most novel and physics-motivated, but discretization
   approximation on irregular mesh introduces uncertainty. Worth trying after simpler ideas.
