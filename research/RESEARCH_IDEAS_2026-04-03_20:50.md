## Ranked Hypotheses

### 1. SWAD — Stochastic Weight Averaging Densely

**Priority: HIGH. Infrastructure exists, flag is `swad=True`, NOT in dead-ends or active experiments.**

SWAD (Cha et al., NeurIPS 2021) averages checkpoints densely from within a flat loss region, rather than averaging two widely-separated endpoints (SWA). The key insight: checkpoints collected during a stable validation window form a tighter ensemble in weight space, landing in a flatter basin with better generalization. Distinct from snapshot ensemble (failed #1807 — snapshots were collected at widely separated cyclical restarts, not from a dense stable window).

The existing `swad` infrastructure in `train.py` tracks `swad_initial_val`, `swad_prev_val`, `swad_checkpoints`, `swad_collecting`, `swad_done`. The collection window triggers when validation stops improving. This is fully implemented.

**Why it might help:** Our ensemble strategy already shows variance reduction is the biggest lever. SWAD gives a single model the benefit of parameter-space averaging without needing separate seeds. The flat-basin argument aligns with why Lion (sign-based, low-curvature updates) works well here — we are already in a geometry where SWAD should thrive.

**Experiment design:**
```
--swad True
--scheduler_type sequential  (standard cosine, no changes)
--seed <single seed>
```
Run with seed=42 (matches batch 1 so we have a direct comparison). SWAD should be compared against the single-seed p_in for the same seed, not the ensemble.

**Key paper:** Cha et al., "SWAD: Domain Generalization by Seeking Flat Minima", NeurIPS 2021. Dense averaging from validation-plateau window. Not to be confused with SWA (uniform averaging of two endpoints).

**Implementation note:** The existing code collects checkpoints when `val_loss < swad_initial_val * 1.01` (approximately). The collection window hyperparameter and the start epoch may need tuning. If the default window is too narrow, SWAD degenerates to a single checkpoint. Recommend also trying a fixed collection window: epochs 140–160 (post-EMA-start, near training end).

**Confidence:** Strong evidence from domain generalization literature. No CFD-surrogate validation yet, but the mechanism (flat basin averaging) is orthogonal to Lion/EMA.

### 2. SGDR Warm Restarts with T_mult=2

**Priority: HIGH. Fully implemented (`scheduler_type="warm_restarts"`), NOT in dead-ends, probably untested.**

Cosine annealing with warm restarts (Loshchilov & Hutter, ICLR 2017) drives the LR to near-zero at each restart, causing the model to escape local minima and explore new basins between restarts. With `T_mult=2`, each cycle is twice as long as the previous, giving a budget of ~10+20+40+80=150 epochs before the full T_max=160 budget is consumed.

The existing flag `scheduler_type="warm_restarts"` with `cosine_T_0=50`, `cosine_T_mult=2`, `cosine_eta_min` is fully wired to `CosineAnnealingWarmRestarts`. Note: frieren #2086 uses SAM only (standard cosine), not SGDR — these are orthogonal.

**Why it might help:** Our current loss curve with cosine annealing has a single descent trajectory. Restarts may help the model find a better basin, especially for the tandem-foil OOD split (p_tan=29.1 is significantly worse than p_in=12.2, suggesting the model overfits to the single-foil geometry and may benefit from more exploration).

**Experiment design:**
```
--scheduler_type warm_restarts
--cosine_T_0 40
--cosine_T_mult 2
--cosine_eta_min 1e-6
--seed 42
```
With T_0=40 and T_mult=2, restarts occur at epochs 40, 120 — well within T_max=160. Compare to standard cosine seed 42.

**Implementation note:** EMA starts at epoch 140 (ema_start_epoch). With T_0=40 and T_mult=2, the LR is near-zero at epoch 120, then ramps back up — EMA averaging will catch the descent into the final basin. This is actually favorable: restarts encourage exploration, EMA stabilizes the final basin.

**Confidence:** Strong from original SGDR paper and widespread use. The T_0/T_mult tuning is empirical — the values above are a principled starting point but a sweep may be warranted.

### 3. Learnable Asinh Scale (per-channel)

**Priority: HIGH. ~2-line code change, directly targets pressure transform quality, genuinely novel.**

The current asinh pressure transform uses `asinh_scale: float = 0.75` — a fixed constant chosen by grid search. The compression function `asinh(p * s)` has different effects at different pressure magnitudes: s too large over-compresses high-pressure peaks, s too small under-compresses them. For tandem foils, the pressure distribution varies significantly between inner and outer channels, suggesting the optimal scale may differ by geometry.

Making `asinh_scale` a learnable `nn.Parameter` initialized to 0.75 allows gradient descent to find the optimal compression jointly with the model weights. The change is:

```python
# In Transolver.__init__:
self.asinh_scale = nn.Parameter(torch.tensor(cfg.asinh_scale))

# In preprocessing (or forward):
p_transformed = torch.asinh(p * self.asinh_scale)
```

A soft-positive constraint (e.g., `F.softplus(self.asinh_scale)`) prevents the scale from going negative.

**Why it might help:** The fixed s=0.75 was found to improve p_oodc by 4-6% (PR #2054). A learned scale can adapt per run to the actual pressure distribution, potentially squeezing out additional improvement — especially for p_tan (OOD tandem configuration where pressure patterns differ).

**Extension worth trying:** Two separate learnable scales — one for positive pressure, one for negative (asymmetric asinh). This directly addresses the physical asymmetry of pressure distributions around airfoils.

**Confidence:** Moderate. Learnable normalization parameters (e.g., learned batch norm statistics) are standard in deep learning. The asinh-specific version is novel here but well-motivated.

### 4. Multi-Scale Output Supervision via Intermediate Features

**Priority: MEDIUM-HIGH. Model already exposes `hidden` (fx_deep), no new flag needed, ~15 lines of code.**

The model's forward pass saves `fx_deep = fx` just before the last TransolverBlock, which is then available in the output dict as `model_out["hidden"]`. This is an intermediate spatial representation at resolution `[B, N, n_hidden]`.

Adding an auxiliary head that predicts the output targets from `fx_deep` (with a lower weight, e.g., 0.1–0.2) creates a deep supervision signal. This forces earlier layers to produce more useful intermediate representations, reducing vanishing gradient effects.

Deep supervision is proven effective in:
- nnUNet (Isensee et al.) — auxiliary losses at each decoder scale
- FNO variants (multi-scale physics operators)
- HED (Holistically-nested Edge Detection) — the foundational architecture

**Experiment design:**
```python
# Auxiliary head (2-layer MLP, same as refine_head but smaller)
aux_head = nn.Sequential(nn.Linear(n_hidden, n_hidden//2), nn.GELU(), nn.Linear(n_hidden//2, 3))

# In training loop:
aux_pred = aux_head(model_out["hidden"])  # [B, N, 3]
aux_loss = 0.15 * F.l1_loss(aux_pred[surface_mask], targets[surface_mask])
loss = main_loss + aux_loss
```

The surface mask should be applied so the auxiliary head is supervised on the most important nodes.

**Why it might help:** The 3-block Transolver is relatively shallow. Deep supervision gives the pre-final-block representations a direct gradient signal from the surface targets, which may improve the quality of the physics attention in the intermediate blocks.

**Implementation note:** The auxiliary head should NOT be used at inference time (predictions come from the final layer only). This is purely a training-time signal. Gradient flow through `fx_deep` is already active since it's computed before `.detach()` is called anywhere.

**Confidence:** Moderate-strong. Deep supervision reliably helps in segmentation and dense prediction tasks. The transfer to mesh-based surrogate models is plausible but not yet validated in literature.

### 5. Asinh Transform on Velocity Channels

**Priority: MEDIUM. Currently pressure-only; wake velocity has similar concentration issues.**

The current asinh transform applies only to the pressure channel (index 2): `_yp[:, :, 2:3] = torch.asinh(_yp[:, :, 2:3] * cfg.asinh_scale)`. The velocity channels (Ux at index 0, Uy at index 1) are left untransformed.

However, velocity in CFD wake flows also has highly non-Gaussian distributions: near-surface boundary layers have very steep gradients; the far-field has near-zero velocity. Uy (cross-stream velocity) in particular can have strong antisymmetry around the chord line. Asinh compression on velocity could reduce the effective dynamic range and make the L1 loss less dominated by high-velocity regions.

**Experiment design:**
```
--asinh_velocity True   (new flag, applies asinh to Ux/Uy channels with separate scale)
--asinh_velocity_scale 0.5  (start smaller than pressure scale=0.75 since velocity range differs)
```

Or simpler: apply the same asinh_scale to all 3 channels and compare.

**Why it might help:** If the velocity residuals in high-gradient wake regions dominate the loss, asinh compression will redistribute gradient weight more uniformly across the mesh. This is the same argument that motivated the pressure asinh in the first place — the velocity channel just hasn't received the same treatment.

**Confidence:** Moderate. The mechanism is sound but velocity has fundamentally different physics from pressure (pressure is scalar, instantaneous; velocity is a vector field with directional structure). The optimal scale may differ significantly from pressure.

### 6. Test-Time Augmentation (TTA) via AoA Perturbation

**Priority: MEDIUM. Inference-only, zero training cost, exploits physical flow smoothness.**

The `aug="aoa_perturb"` flag exists for training-time augmentation (adds small random noise to angle of attack). TTA at inference is different: for a given test sample with AoA=alpha, we also evaluate the model at AoA={alpha-delta, alpha, alpha+delta} and average the three predictions.

Physical motivation: CFD solutions vary smoothly with AoA. The model may have slightly different errors at neighboring AoA values that cancel on average. This is a form of variance reduction that costs 3x inference time but zero training time.

**Experiment design:**
```python
# At inference, for each test sample:
aoa_deltas = [-1.0, 0.0, +1.0]  # degrees
preds = []
for delta in aoa_deltas:
    modified_x = x.clone()
    modified_x[:, aoa_feature_idx] += delta * aoa_scale  # adjust AoA feature
    preds.append(model(modified_x)["preds"])
final_pred = torch.stack(preds).mean(0)
```

The AoA feature index must be identified in the input encoding (24-dim x from `prepare_multi.py`). The delta magnitude should be small enough that the perturbed AoA is physically plausible.

**Implementation note:** This requires understanding which input feature encodes AoA. From `data/prepare_multi.py` (read-only), the 24-dim input includes geometry coordinates plus flow condition scalars. The AoA feature may need to be re-normalized after perturbation. This is the trickiest part to implement correctly.

**Confidence:** Moderate. TTA via input perturbation is well-established in image tasks (flips, crops). The physics-based analog here is sound in principle, but the feature-space perturbation must correctly represent a flow condition change.

### 7. SGDR + SWAD Combined (Snapshot-Averaging from Restarts)

**Priority: MEDIUM. Combines ideas 1 and 2 for a principled checkpoint diversity approach.**

Run SGDR warm restarts (scheduler_type="warm_restarts") and collect SWAD checkpoints from within the last (longest) cosine cycle only. With T_0=40, T_mult=2, the last cycle spans epochs 120–160. SWAD collects checkpoints in the stable region of this final descent.

The rationale: SGDR explores different basins across restarts, but SWAD averaging from only the final basin gives the stability benefit without averaging across different minima (which could hurt).

This is distinct from snapshot ensemble (#1807, failed) which averaged across different restarts rather than within one stable descent.

**Experiment design:**
```
--scheduler_type warm_restarts
--cosine_T_0 40
--cosine_T_mult 2
--swad True
--swad_start_epoch 130   (start SWAD collection only in final cycle)
```

**Confidence:** Speculative. This combination hasn't been tested and requires careful implementation to avoid averaging across restart boundaries.

### 8. HyperP — Hypernetwork-based Parameter Generation (arXiv 2603.28743)

**Priority: LOW-MEDIUM. On the user's explicit wishlist from issue #1926. Most exploratory.**

HyperP uses a hypernetwork to generate model parameters conditioned on input context (e.g., flow conditions, geometry statistics). For the tandem foil problem, the hypernetwork could condition the main model's attention parameters on the Reynolds number and AoA, potentially giving better adaptation to OOD flow regimes (p_tan=29.1 is by far the worst metric, suggesting the OOD regime is poorly handled).

This is architecturally the most complex idea on this list. The human researcher team flagged it in issue #1926 as "exploratory" alongside MSA and mHC (which were deprioritized). The architecture modification overhead is high and the risk of EMA/Lion absorbing the change (as with other architectural changes) is real.

**Suggested approach if assigning this:** Start with a tiny hypernetwork (2-layer MLP) that generates only the `condition` scalar used in AdaLayerNorm, conditioned on (re, aoa, tandem_flag). The `condition` is already used in the final TransolverBlock. This is the minimal footprint that tests the hypernetwork concept.

**Confidence:** Low. High complexity, mechanism may be absorbed by existing adaptation via condition vectors. Consider only after simpler ideas are exhausted.
